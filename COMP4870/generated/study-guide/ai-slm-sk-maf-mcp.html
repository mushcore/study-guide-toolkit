The Big Picture

Everything below is one tech stack. The boxes rhyme — don't confuse SDK (SK / MAF) with protocol (MCP) with model (SLM).

Rule of thumb SK is a library. MAF is a higher-level library (agents-oriented). MCP is a wire protocol — a contract any language can speak. SLM is a runtime artifact — the model file that does the inference.

A · SLM — Small Language Models

Compact LLMs (millions to ~10B parameters) that run on your laptop. Lower latency, privacy, offline, edge deployment.

SLM vs LLM

SLMLLM
Paramsmillions → ~10Bhundreds of billions → trillions
Runs onlaptop / phone / edgecloud data centre
Latencylownetwork-bound
Privacystays on-devicesent to provider
Costfree (compute only)pay per token
Good atnarrow/tuned tasksbroad general reasoning

Slides emphasise: SLMs focus on doing fewer things well. LLMs may have trillions of parameters; SLMs are compact versions.

How SLMs are made smaller

Knowledge distillation

A smaller "student" model learns from a larger "teacher" model.

Pruning

Removes redundant or less impactful parameters.

Quantization

Reduces numerical precision to save memory and compute.

Architecture tweaks

Streamlined layers and tokenization strategies for faster inference.

The three ways to get a model (from class)

1. Ollama (local)

Download, run, and serve SLMs from localhost:11434. Used in Demo #1 with Phi-3.

2. GitHub Models (cloud)

Inference endpoint https://models.github.ai/inference. Auth via GitHub Personal Access Token (PAT). Models like openai/gpt-4o-mini.

3. Docker Model Runner / AI Toolkit

Pull SLMs via Docker Desktop or VS Code AI Toolkit. OpenAI-compatible endpoint at http://localhost:5272/v1/ or :12345.

Ollama commands

bash 
# Download Ollama: https://www.ollama.com/download

ollama pull phi3:latest       # download a model
ollama list                   # list local models (alias: ollama ls)
ollama show phi3:latest       # inspect a model
ollama run phi3               # interactive chat (type /bye to quit)
# Ollama HTTP API is served at http://localhost:11434/

Docker Model Runner commands

bash 
docker model list
docker model pull ai/ministral3:latest
docker model run llama3.2

# Enable TCP runner (serves OpenAI-compatible API):
docker desktop enable model-runner --tcp=12345
# Then:  http://localhost:12345/engines/llama.cpp/v1
Why SLMs Privacy (data never leaves device), cost (no per-token fees), offline (works without internet), edge deployment, and — per slides — sustainable, customizable alternatives to LLMs. Use cases: mobile assistants, customer support bots, real-time translation, domain-specific tools (healthcare, finance).

B · Semantic Kernel (SK)

Quote from slide 23: "Semantic Kernel is an open-source SDK that lets you easily build agents that can call your existing code. As a highly extensible SDK, you can use Semantic Kernel with models from OpenAI, Azure OpenAI, Hugging Face, and more!"

Key facts (slide 24)

  • Works with C#, Python, Java.
  • Plugins — using prompts and native functions.
  • Context variables.
  • Connectors — using memory and models.
  • Planners — AI orchestrator.

Source: https://learn.microsoft.com/en-us/semantic-kernel/overview/

NuGet packages

bash 
# Low level libraries
dotnet add package OpenAI
dotnet add package Azure.AI.OpenAI

# Higher level libraries
dotnet add package Microsoft.SemanticKernel

Core concepts

ConceptWhat it is
KernelThe main orchestrator that wires a model + plugins + memory.
KernelBuilder / AddKernel()Fluent builder to assemble the kernel and its providers.

SK bootstrap — GitHub Models

This is the W02 lab pattern (Labs/W02 AI on Github/GitHubAiModelSK/Program.cs):

C# 
using Microsoft.SemanticKernel;
using OpenAI;
using System.ClientModel;

var builder = WebApplication.CreateBuilder(args);

// Read from appsettings.json
var modelId   = builder.Configuration["AI:Model"]!;
var uri       = builder.Configuration["AI:Endpoint"]!;
var githubPAT = builder.Configuration["AI:PAT"]!;

var client = new OpenAIClient(
    new ApiKeyCredential(githubPAT),
    new OpenAIClientOptions { Endpoint = new Uri(uri) });

builder.Services.AddOpenAIChatCompletion(modelId, client);
builder.Services.AddKernel();

var app = builder.Build();
// … MVC / Razor Pages wiring
appsettings.json (GitHub Models) Model: openai/gpt-4o-mini · Endpoint: https://models.github.ai/inference · Token: your GitHub PAT.

Prompt-based function — skprompt.txt

Prompt-based kernel functions live in a folder named after the function, containing skprompt.txt and (optionally) config.json. Inputs use {{$input}} template syntax.

MAF with Ollama (full)

using Microsoft.Agents.AI;
using Microsoft.Extensions.AI;
using OllamaSharp;

string ollamaUrl = "http://localhost:11434/";
string modelId   = "llava:7b";
IChatClient ollamaClient = new OllamaApiClient(new Uri(ollamaUrl), modelId);

ChatClientAgent agent = new ChatClientAgent(
    ollamaClient,
    new ChatClientAgentOptions
    {
        Name = "Astronomer",
        ChatOptions = new ChatOptions
        {
            Instructions = "You are an astronomer who specializes in the solar system...",
            Temperature  = 0.7f
        }
    }
);

Console.WriteLine(await agent.RunAsync("How far is earth from the sun?"));

// Streaming variant:
await foreach (var response in agent.RunStreamingAsync("Which planet is closest to the sun?"))
    Console.Write(response);

Turning a C# method into an AITool

This is the MAF equivalent of SK's [KernelFunction]:

C# 
public class WeatherService
{
    [Description("Get the weather details of a location using city")]
    public static async Task<string> GetWeatherAsync(
        [Description("The city for which the weather report is needed")] string city)
    {
        /* call openweathermap.org */
    }
}

// Wrap as AITool:
AITool getWeatherTool = AIFunctionFactory.Create(
    WeatherService.GetWeatherAsync,
    name: "GetWeather",
    description: "Get the weather details of a location using city name");

// Pass tools to the agent:
AIAgent agent = chatClient.AsAIAgent(
    name: "WeatherPoetAgent",
    instructions: "You are a helpful assistant...",
    tools: [getWeatherTool, /* more tools */]);

MAF Planner pattern (Demo #3)

Planner intuition You register multiple tools (GetCityNameFromJson, GetWeather, WriteShortPoem). The agent's instructions describe when to call each. The model plans the call order itself — you never hardcode "first call GetCity then call GetWeather"; the LLM figures it out from descriptions.

Multi-turn with sessions

C# 
AgentSession session = await agent.CreateSessionAsync();

while (true) {
    var input = Console.ReadLine();
    if (string.IsNullOrWhiteSpace(input)) break;

    await foreach (var update in agent.RunStreamingAsync(input, session))
        if (!string.IsNullOrEmpty(update.Text)) Console.Write(update.Text);
}
SK vs MAF — don't conflate SK uses Kernel / [KernelFunction] / ImportPluginFromType. MAF uses AIAgent / AIFunctionFactory.Create / AsAIAgent(). Both can use the same model endpoints, but the API surface differs. Elmasry will test this.

D · Model Context Protocol (MCP)

An open protocol that lets any AI agent discover and call external tools — regardless of the SDK or language. Think "USB for AI tools."

MCP primitives

PrimitivePurposeIn class
ToolsCallable functions the AI can invoke (verbs).W05 GetBeveragesByOrigin, W06 get_games_by_country
ResourcesRead-only data blobs the AI can fetch (nouns).not demoed in labs — mentioned conceptually
PromptsPre-written prompt templates the server exposes.not demoed in labs — mentioned conceptually

Mnemonic: TRP — Tools, Resources, Prompts.

Client ↔ Server

MCP Server

Exposes tools/resources/prompts. In .NET: a console app (AddMcpServer()) or an Azure Function ([McpToolTrigger]).

MCP Client

An AI app that McpClient.CreateAsync(transport), calls ListToolsAsync(), then feeds the tools into a chat model's function-calling.

Transports

TransportWhenLab
stdio Local. Client launches the server as a child process and talks over stdin / stdout. W05 Beverages — console MCP server, WithStdioServerTransport()
HTTP / SSE (Streamable HTTP) Remote. Hosted as an endpoint (e.g., Azure Functions). Uses Server-Sent Events for streaming. W06 FIFA — Azure Functions, endpoint /runtime/webhooks/mcp

W05 — stdio MCP server

C# (Program.cs) 
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;

var builder = Host.CreateApplicationBuilder(args);

builder.Logging.AddConsole(opts =>
    opts.LogToStandardErrorThreshold = LogLevel.Trace);

builder.Services
    .AddMcpServer()
    .WithStdioServerTransport()
    .WithToolsFromAssembly();

await builder.Build().RunAsync();
C# (BeverageTools.cs) 
using System.ComponentModel;
using ModelContextProtocol.Server;

[McpServerToolType]
public static class BeverageTools
{
    private static readonly BeverageService _svc = new();

    [McpServerTool, Description("Get beverages that contain sugar in their main ingredient")]
    public static string GetBeveragesContainingSugar()
    {
        var beverages = _svc.GetBeveragesContainingSugar().GetAwaiter().GetResult();
        return JsonSerializer.Serialize(beverages, BeverageContext.Default.ListBeverage);
    }

    [McpServerTool, Description("Get beverages that originate from a specific country or region")]
    public static string GetBeveragesByOrigin(
        [Description("The origin country or region to filter beverages by")] string origin)
    {
        // …
    }
}

W05 — MCP client using GitHub Models

C# 
using ModelContextProtocol.Client;
using Microsoft.Extensions.AI;

// 1. launch the server as a child process over stdio
var clientTransport = new StdioClientTransport(new StdioClientTransportOptions
{
    Name = "Beverages MCP Server",
    Command = "dotnet",
    Arguments = new[] { "run", "--project", "../BeveragesMcpServer/BeveragesMcpServer.csproj" }
});

// 2. create the MCP client
var client = await McpClient.CreateAsync(clientTransport);

// 3. discover tools
var mcpTools = await client.ListToolsAsync();

// 4. wire tools into a chat client (GitHub Models)
var openAIClient = new OpenAIClient(new ApiKeyCredential(githubToken!),
    new OpenAIClientOptions { Endpoint = new Uri("https://models.inference.ai.azure.com") });
var baseChatClient = openAIClient.GetChatClient(modelName).AsIChatClient();
var chatClient = new ChatClientBuilder(baseChatClient)
    .UseLogging(loggerFactory)
    .UseFunctionInvocation()   // enables auto tool-calling
    .Build();

// 5. feed tools into the chat request
var response = await chatClient.GetResponseAsync(messages,
    new ChatOptions { Tools = mcpTools.ToArray<AITool>() });

W06 — Azure Functions MCP server (HTTP/SSE)

C# (FifaTools.cs) 
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Extensions.Mcp;

public class FifaTools
{
    private static readonly FifaService _fifaService = new();

    [Function(nameof(GetGamesByCountry))]
    public string GetGamesByCountry(
        [McpToolTrigger("get_games_by_country",
            "Get FIFA World Cup games played in a specific country and return as JSON")]
        ToolInvocationContext context)
    {
        var country = context.Arguments?.GetValueOrDefault("country")?.ToString() ?? "";
        var games = _fifaService.GetGamesByCountry(country).GetAwaiter().GetResult();
        return JsonSerializer.Serialize(games, FifaWorldCupContext.Default.ListFifaWorldCup);
    }
}
C# (Program.cs) 
using Microsoft.Azure.Functions.Worker.Builder;

var builder = FunctionsApplication.CreateBuilder(args);
builder.ConfigureFunctionsWebApplication();
builder.EnableMcpToolMetadata();

builder.ConfigureMcpTool("get_games_by_country")
    .WithProperty("country", "string", "The country to filter games by", false);

builder.ConfigureMcpTool("get_games_by_city")
    .WithProperty("city", "string", "The city to filter games by", false);

builder.Build().Run();

E · Integration & Bigger Picture

How the five pieces fit together in one real .NET app (it's simpler than it looks).

The common shape

Agent (MAF AIAgent) → IChatClient (SK or MAF) → Model (Ollama / GitHub / Docker)
Tools (AITool) ← AIFunctionFactory.Create OR McpClient.ListToolsAsync

The agent's planner decides when to call tools. Tools come from either in-process C# (AIFunctionFactory) or out-of-process MCP servers (McpClient.ListToolsAsync) — both return AITool instances.

Who owns what

LayerOwnerExample
Model inferenceSLM runtimeOllama server at :11434, GitHub Models API, Docker Model Runner at :12345
Chat client abstractionMicrosoft.Extensions.AIIChatClient, AsIChatClient()
Agent wrappingMAF (or SK)AIAgent, AsAIAgent(), ChatClientAgent
Tool packagingAIFunctionFactory (MAF-style) or [KernelFunction] (SK-style)Both produce things the agent can call
Cross-process tool contractMCPstdio or HTTP; discovered via ListToolsAsync()
Orchestration / hostingAspire (when multiple apps)Aspire AppHost can launch both MCP server and client projects
Assignment 2 framing When the assignment says "add AI to your app," the common stack is: SK or MAF as the SDK, GitHub Models or Ollama as the model, [KernelFunction] or AIFunctionFactory.Create for tools. For tools that should be reusable across apps (or across languages), expose them via an MCP server.

When to pick which

Use SK when…

You want planners, memory connectors, or you're following older Microsoft docs. Prompt-based functions via skprompt.txt are SK's speciality.

Use MAF when…

You want a simple AIAgent + tools story. Simpler API: chatClient.AsAIAgent(...) + pass tools: [...].

Use MCP when…

Your tools should be consumable by multiple clients (VS Code, Claude, other agents). Or when tools run in a different process / language.

Common gotchas / exam traps

SK vs MAF namespace SK's agent types live in Microsoft.SemanticKernel. MAF's AIAgent lives in Microsoft.Agents.AI. Not interchangeable.
[KernelFunction] vs AIFunctionFactory [KernelFunction] = SK idiom. AIFunctionFactory.Create = MAF idiom. They serve the same purpose from different SDKs.
skprompt folder rule skprompt.txt and config.json must live in a folder whose name matches the function name. Put them in the wrong folder and SK won't find the function.
Ollama model name format Models use name:tag (e.g., phi3:latest, phi3:mini, llava:7b). Missing tag may default to latest.
AsAIAgent location AsAIAgent(...) is an extension on IChatClient / ChatClient from Microsoft.Agents.AI. You call it on the chat client, not on the agent.
W06 requires Azurite The Azure Functions MCP extension uses Azure Storage queues for SSE transport. Without Azurite running locally, func start will fail. Also pin Microsoft.Azure.Functions.Worker = 2.0.0.
MCP tool attribute differs by host Console server: [McpServerToolType] + [McpServerTool]. Azure Functions: [McpToolTrigger(name, description)]. Different NuGet, different attribute.
GitHub PAT token usage GitHub Models accepts your regular GitHub PAT with no special scopes. Wrap it in new ApiKeyCredential(token) or new AzureKeyCredential(token) depending on which client you use.
UseFunctionInvocation() On the MCP client, you must call new ChatClientBuilder(base).UseFunctionInvocation().Build() for the model to actually invoke tools. Without it, the model only suggests tool calls.
Instructions matter The agent's instructions string is how the LLM knows when to call each tool. Describe the order and triggers there — you can't hardcode "call A then B" in C#.

Self-interrogation prompts

Reveal-on-click. Do these closed-book first. If you can answer 6/8 out loud, you're exam-ready on this topic.

1. Without looking, write the minimum C# needed to chat with Ollama via Semantic Kernel.

Wire OllamaSharp's OllamaApiClient as an IChatClient, build a ChatHistory, and call GetStreamingResponseAsync. Or use the higher-level SK pattern of AddOpenAIChatCompletion pointing at Ollama's OpenAI-compatible endpoint. The Ollama URL is http://localhost:11434/.

2. Explain in two sentences why MCP exists.

Before MCP, every agent SDK had its own way to register tools — tools built for SK weren't callable from Claude Desktop, VS Code Copilot, or a different framework. MCP is an open wire protocol: any MCP client (regardless of SDK) can discover and call tools from any MCP server, over stdio or HTTP.

3. What's the difference between [KernelFunction] and AIFunctionFactory.Create?

Both expose a C# method to an AI model. [KernelFunction] is Semantic Kernel's attribute — you import the type via ImportPluginFromType<T>. AIFunctionFactory.Create is Microsoft Agent Framework's runtime factory — it returns an AITool you pass via tools: [...] on the agent.

4. An MCP server has 5 tools. Describe end-to-end how a user prompt results in one of those tools being called.

Client calls McpClient.CreateAsync(transport), then ListToolsAsync() returns the 5 tool definitions. Client wraps them via ChatOptions { Tools = mcpTools.ToArray<AITool>() } and feeds the user prompt to a chat model configured with .UseFunctionInvocation(). The model returns a tool-call request; the chat client invokes the MCP tool over the transport; result is fed back as a tool message; the model produces the final assistant reply.

5. Why do W05 and W06 use different attributes for the same purpose ([McpServerTool] vs [McpToolTrigger])?

Because the hosting model is different. W05 hosts MCP as a console process, so it uses the ModelContextProtocol package and the [McpServerToolType]/[McpServerTool] attributes. W06 hosts MCP inside Azure Functions, so it uses Microsoft.Azure.Functions.Worker.Extensions.Mcp and its [McpToolTrigger] attribute, which the Functions runtime recognises as a trigger.

6. If you add a new tool to an SK agent, what must the tool's C# method have?

A [KernelFunction] attribute, typically alongside [Description("…")] on the method and on each parameter. Then call kernel.ImportPluginFromType<YourPlugin>().

7. Name three reasons to prefer an SLM over an LLM.

Privacy (data stays on device), cost (no per-token fees), offline/edge capability (no internet required), lower latency, and customisation (you can fine-tune small models cheaply).

8. How does the MAF planner decide when to call which tool?

It doesn't plan statically. You register tools with descriptive [Description(...)] attributes and give the agent instructions that describe when to use them. The LLM itself — via function calling — chooses which tool to call from the descriptions and arguments in the model's tool-choice output.

9. W05's MCP client code calls UseFunctionInvocation(). What does that unlock?

It enables automatic tool calling: when the model responds with a "call this tool" request, the chat client invokes the MCP tool, feeds the result back into the conversation, and loops until the model produces a plain assistant message. Without it, the client would surface the tool call to you and stop.
