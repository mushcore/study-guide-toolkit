## 1. FULL TAXONOMY OF LINUX IPC MECHANISMS

### 1.1 System V IPC vs POSIX IPC Overview

| Mechanism | System V | POSIX | Process Type |
|-----------|----------|-------|--------------|
| **Pipes** | N/A | N/A | Related processes only |
| **Named Pipes (FIFO)** | mkfifo, mknod | N/A | Related & unrelated |
| **Message Queues** | msgget, msgsnd, msgrcv | mq_open, mq_send, mq_receive |  Both |
| **Shared Memory** | shmget, shmat, shmdt | shm_open, mmap | Both |
| **Semaphores** | semget, semop, semctl | sem_open, sem_wait, sem_post | Both |
| **Signals** | signal, kill, raise | sigaction | Both |
| **Sockets** | socket, bind, listen | socket, bind, listen | Unrelated processes |

### 1.2 Pipes (Unnamed Pipes)
- **Used for:** Communication between related processes (parent-child)
- **Directionality:** Unidirectional (one-way communication)
- **Lifespan:** Process lifetime (kernel-managed buffer)
- **System Call:** `pipe()`
- **File Descriptors:** pipe_fds[0] = read end, pipe_fds[1] = write end
- **Bidirectional Communication:** Requires TWO pipes (one in each direction)

### 1.3 Named Pipes (FIFO)
- **Used for:** Communication between unrelated processes
- **Directionality:** Bidirectional
- **Lifespan:** Exists in filesystem until explicitly deleted
- **Creation Commands:** 
  - `mkfifo [OPTION] ... NAME ...` (creates named pipes)
  - `mknod [OPTION] ... NAME TYPE [MAJOR MINOR]` (creates special files)
- **Location:** Filesystem (device file: prw-r--r--)
- **Persistence:** Kernel lifetime (exists until removed)

### 1.4 Shared Memory
- **File Mapping:** Reading/writing to mapped memory areas causes file read/write
- **Anonymous Mapping:** Like malloc(), contents initialized to zero
- **Shared Between Processes:** Via `shmat()` attachment
- **System V API:** shmget, shmat, shmdt, shmctl
- **POSIX API:** shm_open, ftruncate, mmap, munmap

### 1.5 Message Queues
- **Purpose:** Exchange data in discrete units called messages
- **Each Message:** Can have associated priority
- **System V API:** msgget, msgsnd, msgrcv, msgctl
- **POSIX API:** mq_open, mq_send, mq_receive, mq_notify
- **Persistence:** Kernel lifetime

### 1.6 Semaphores
- **Named Semaphores:** Identified by name (e.g., /my_semaphore), synchronize multiple processes
- **Unnamed Semaphores (Memory-based):** Used to synchronize threads within same process
- **System V API:** semget, semop, semctl
- **POSIX API:** sem_open, sem_wait, sem_post, sem_close, sem_unlink

### 1.7 Signals
- **Definition:** Software interrupt (event) generated by system or user
- **Uses:** Asynchronous notification of events
- **Handlers:** SIG_IGN (ignore), SIG_DFL (default action), or user-defined function
- **Examples:** SIGUSR1, SIGUSR2, SIGINT (Ctrl+C), SIGSTOP (Ctrl+Z)

### 1.8 Sockets
- **Purpose:** Communication between related AND unrelated processes
- **Directionality:** Bidirectional
- **Transport:** TCP (connection-oriented, reliable) or UDP (connectionless, unreliable)
- **Domains:** AF_UNIX (local), AF_INET (IPv4), AF_INET6 (IPv6)
- **Types:** SOCK_STREAM (TCP), SOCK_DGRAM (UDP)

### 2.3 SHARED MEMORY

#### mmap() - Map or unmap files into memory
```c
#include <sys/mman.h>

void *mmap(void *addr, size_t length, int prot, int flags,
           int fd, off_t offset);
int munmap(void *addr, size_t length);
```

**Parameters:**
- `addr`: User-specified or kernel-generated address (addr = NULL)
- `length`: Size of mapping in bytes
- `prot`: Protection (PROT_NONE, PROT_READ, PROT_WRITE, PROT_EXEC)
- `flags`: MAP_SHARED or MAP_PRIVATE
- `fd`: File descriptor identifying file to be mapped
- `offset`: Starting point of file (offset = 0 if entire file is mapped)

**Returns:** Starting address of mapping on success, MAP_FAILED on error

## 3. WHEN TO USE EACH IPC MECHANISM (TRADEOFFS)

| Mechanism | Best For | Advantages | Disadvantages |
|-----------|----------|-----------|----------------|
| **Pipes** | Related processes, simple producer-consumer | Lightweight, easy, kernel-managed | Unidirectional only, related processes only |
| **Named Pipes** | Unrelated processes, simple IPC | File-based, bidirectional, filesystem visibility | Still unidirectional per file, limited buffering |
| **Shared Memory** | High-performance, large data structures | Fast (no copying), flexible, large capacity | Synchronization required, complex memory management |
| **Message Queues** | Structured messages, priority handling, multiple consumers | Structured data, priority support, decoupled | More overhead than pipes, limited message size |
| **Semaphores** | Synchronization, mutual exclusion | Efficient, flexible, can synchronize across processes | Complex to understand, error-prone if not careful |
| **Signals** | Asynchronous events, notifications | Lightweight, immediate | Limited data transfer, not suitable for bulk data |
| **Sockets** | Unrelated processes, network communication, bidirectional | Bidirectional, network-capable, standard protocol | More overhead, complexity for local IPC |

## 5. MAP_SHARED FLAG & MMAP USE CASES

### MAP_SHARED vs MAP_PRIVATE
- **MAP_SHARED:** Changes visible to other processes mapping same file (shared memory)
- **MAP_PRIVATE:** Changes private to calling process; original file unchanged

### mmap() Use Cases
1. **File Mapping:** Map file contents into memory for efficient access
2. **Anonymous Mapping:** Allocate memory like malloc(), zero-initialized
3. **Shared Memory:** Multiple processes access same memory region
4. **Device Access:** Map device memory into process address space

## 6. CODE EXAMPLES (VERBATIM FROM SLIDES)

### Example: Pipe Parent-Child Communication

```c
#include <stdio.h>
#include <unistd.h>

#define BUFFER_SIZE 256

int main() {
    int pipefd[2];
    char write_messages[2][30] = {
        "Hello from user 1",
        "Hello from user 2"
    };
    char read_message[30];
    
    int return_status = pipe(pipefd);
    printf("reading file descriptor : %d\n", pipefd[0]);
    printf("writing file descriptor : %d\n", pipefd[1]);

    if (return_status == -1) {
        printf(" Pipe creation is failed.\n");
        return 1;
    }

    // MSG 1
    printf("Writing to pipe : Msg 1 is %s\n", write_messages[0]);
    write(pipefd[1], write_messages[0], sizeof(write_messages[0]));
    read(pipefd[0], read_message, sizeof(read_message));
    printf("Reading from pipe : Msg 1 is %s\n", read_message);

    // MSG 2
    printf("Writing to pipe : Msg 2 is %s\n", write_messages[1]);
    write(pipefd[1], write_messages[1], sizeof(write_messages[1]));
    read(pipefd[0], read_message, sizeof(read_message));
    printf("Reading from pipe : Msg 2 is %s\n", read_message);

    return 0;
}
```

### Example: Parent-Child Fork with Pipe & dup2

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <wait.h>

#define BUFFER_SIZE 256

int main() {
    int pipefd[2];
    char buffer[BUFFER_SIZE];
    pid_t pid;

    if (pipe(pipefd) == -1) {
        perror("pipe");
        exit(EXIT_FAILURE);
    }

    pid = fork();
    if (pid < 0) {
        perror("fork failed");
        exit(EXIT_FAILURE);
    }

    if (pid == 0) {
        // Child process
        close(pipefd[1]);
        int num_bytes = read(pipefd[0], buffer, BUFFER_SIZE);
        if (num_bytes > 0) {
            printf("Child: Received message from parent: %s\n", buffer);
        }
        close(pipefd[0]);
        exit(EXIT_SUCCESS);
    } else {
        // Parent process
        close(pipefd[0]);
        const char* message = "Hello from parent!";
        size_t num_bytes = write(pipefd[1], message, strlen(message) + 1);
        close(pipefd[1]);
        wait(NULL);
        exit(EXIT_SUCCESS);
    }

    return 0;
}
```

### Example: Memory Mapping File

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>

int main() {
    int fd;
    struct stat file_info;
    char *mapped_file;

    fd = open("myfile.txt", O_RDWR);
    if (fd == -1) {
        perror("open");
        exit(EXIT_FAILURE);
    }

    if (fstat(fd, &file_info) == -1) {
        perror("fstat");
        exit(EXIT_FAILURE);
    }

    off_t file_size = file_info.st_size;
    mapped_file = mmap(NULL, file_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    if (mapped_file == MAP_FAILED) {
        perror("mmap");
        exit(EXIT_FAILURE);
    }

    // Use mapped_file...

    for (off_t i = 0; i < file_size; i++) {
        mapped_file[i] = (mapped_file[i]);
    }

    if (munmap(mapped_file, file_size) == -1) {
        perror("munmap");
        exit(EXIT_FAILURE);
    }

    if (close(fd) == -1) {
        perror("close");
        exit(EXIT_FAILURE);
    }

    return 0;
}
```

## 7. EXAM TRAPS & KEY DISTINCTIONS

### Unidirectional vs Bidirectional
- **Pipes:** Unidirectional ONLY (one-way)
- **Named Pipes:** Can be bidirectional but one direction per file
- **Sockets:** Bidirectional by default
- **socketpair():** Truly bidirectional (2 connected sockets)

### Related vs Unrelated Processes
- **Related:** Parent-child, fork() relationship → use pipes
- **Unrelated:** No fork relationship → use named pipes, message queues, shared memory, sockets, semaphores

### Persistence (Kernel Lifetime vs Process Lifetime)
- **Process Lifetime:** Pipes (kernel buffer managed, destroyed when last reference closes)
- **Kernel Lifetime:** Named pipes (FIFO), message queues, shared memory, semaphores, sockets (persist until explicitly deleted)
- **Filesystem Visible:** Named pipes (FIFO) only

### Critical Trap: pipe() Creates TWO File Descriptors
```c
pipe(pipefd);
// pipefd[0] = READ end
// pipefd[1] = WRITE end
// MUST close unused ends after fork()!
```

### Signal Handlers: SIG_IGN vs SIG_DFL
- **SIG_IGN:** Ignore the signal completely
- **SIG_DFL:** Use system's default action (usually terminate process)
- **Function pointer:** User-defined handler

## 9. PRACTICE QUESTIONS (8-10) - T/F, MCQ, SHORT ANSWER

### Question 1 (True/False)
**Pipes can be used for communication between unrelated processes.**
- **Answer:** FALSE
- **Explanation:** Pipes are only for related processes (parent-child). Use named pipes (FIFOs) for unrelated processes.
- **Source:** Part 8, Slide 7

### Question 2 (Multiple Choice)
**Which of the following is bidirectional?**
A) Unnamed pipes
B) Named pipes (FIFO)
C) Sockets
D) Both B and C

- **Answer:** D (Both B and C)
- **Explanation:** Named pipes can be bidirectional (via read/write in different directions). Sockets are bidirectional by design.
- **Source:** Part 8, Slide 7, 53

### Question 3 (Multiple Choice)
**What does sem_wait() do?**
A) Increment the semaphore
B) Decrement the semaphore and block if value is zero
C) Check semaphore value without modifying
D) Close the semaphore

- **Answer:** B
- **Explanation:** sem_wait() decrements (locks). If value becomes negative, it blocks until sem_post() is called.
- **Source:** Part 8, Slide 40

### Question 4 (Short Answer)
**What are the two parameters to ftok() and what do they represent?**
- **Answer:** pathname (path to file), proj_id (project identifier). ftok() converts these to a System V IPC key.
- **Source:** Part 8, Slide 35

### Question 5 (True/False)
**MAP_SHARED changes to mmap() memory are visible to other processes mapping the same file.**
- **Answer:** TRUE
- **Explanation:** MAP_SHARED enables shared memory across processes.
- **Source:** Part 8, Slide 29

### Question 6 (Multiple Choice)
**Which IPC mechanism has kernel lifetime (persists after process terminates)?**
A) Pipes
B) Message queues
C) Shared memory
D) Both B and C

- **Answer:** D (Both B and C)
- **Explanation:** Message queues and shared memory (System V) persist in kernel until explicitly deleted. Pipes are destroyed when last reference closes.
- **Source:** Part 8, Slides 5-6, 35-36

### Question 7 (Short Answer)
**List the 5 main POSIX semaphore functions.**
- **Answer:** sem_open(), sem_wait(), sem_post(), sem_close(), sem_unlink()
- **Source:** Part 8, Slide 40

### Question 8 (Multiple Choice)
**Which socket type is used for TCP (connection-oriented, reliable)?**
A) SOCK_DGRAM
B) SOCK_STREAM
C) SOCK_RAW
D) SOCK_RDM

- **Answer:** B (SOCK_STREAM)
- **Explanation:** SOCK_STREAM provides sequenced, reliable, two-way byte streams (TCP). SOCK_DGRAM is UDP (datagram).
- **Source:** Part 8, Slide 53

### Question 9 (True/False)
**Signals are suitable for transferring large amounts of data between processes.**
- **Answer:** FALSE
- **Explanation:** Signals are lightweight notifications only. Use pipes, message queues, or shared memory for bulk data transfer.
- **Source:** Part 8, Slide 44-45

### Question 10 (Short Answer)
**What must a parent process do after fork() to establish one-way communication to child via pipe?**
- **Answer:** Close the write end in parent (close(pipefd[1])) and close the read end in child (close(pipefd[0])). Then parent writes to pipefd[1] and child reads from pipefd[0].
- **Source:** Part 8, Slide 15-16
