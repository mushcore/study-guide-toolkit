 Let's summarize by going through some advantages and disadvantages of the proxy pattern. First, you want to use this pattern if you want to execute something before or after the primary service logic. What I mean by this is in our movie example, we wanted to do some checks, some membership checks to see if I'm a moderator or member before I'm sending calls to the movie database. Second is since the proxy itself implements the same interface as a service, the proxy can be passed through any object that's expecting that same service interface. Third is that we're encapsulating and controlling access to an expensive object. In our movie database example, it's an expensive operation to call that database over and over again. Therefore, we're controlling access to who can access the database and at what time? In the fourth reason is that a proxy can work even if the original service is not available. So this is more relating to, let's say in the YouTube example I gave, Google's headquarters servers where the YouTube videos are located goes down. But that can be okay if the Seattle servers have a cached version of all the videos. Because anybody who's trying to access YouTube can then query the Seattle servers instead of the original YouTube servers. And the final example is that the proxy pattern follows the open closed principle where we can introduce new behaviors by adding new proxies instead of modifying an existing service. Some disadvantages of this pattern is we can complicate the code base if there are multiple proxies that exist for the same service. And the second disadvantage is that the response from the service might be delayed if the proxies carrying out a lot of behaviors before calling the service or after invoking the service.
