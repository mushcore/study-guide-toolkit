9. Node API object and CIDR range:

   $ kubectl get no kind-control-plane -o yaml | grep -A2 podCIDR
   spec:
     podCIDR: 10.244.0.0/24
     podCIDRs:
     - 10.244.0.0/24

   a. The CIDR range used for pod IPs is 10.244.0.0/24.

      CIDR stands for Classless Inter-Domain Routing. It is a method for
      allocating IP addresses and routing that replaces the older classful
      (Class A/B/C) addressing system. The /24 notation means the first
      24 bits are the network prefix, leaving 8 bits for host addresses
      (256 addresses, 10.244.0.0 through 10.244.0.255).

17-18. Create the core-k8s pod:

   $ kubectl create -f pod.yaml
   pod/core-k8s created

   Where pod.yaml contains:
     apiVersion: v1
     kind: Pod
     metadata:
       name: core-k8s
       labels:
         role: just-an-example
         app: my-example-app
         organization: friends-of-manning
         creator: jay
     spec:
       containers:
       - name: any-old-name-will-do
         image: docker.io/busybox:latest
         command: ['sleep','10000']
         ports:
         - name: webapp-port
           containerPort: 80
           protocol: TCP

   $ kubectl get po
   NAME                     READY   STATUS    RESTARTS   AGE
   core-k8s                 1/1     Running   0          15s
   nginx-5d59d67564-abc12   1/1     Running   0          5m
   ...

   $ kubectl get po core-k8s -o yaml

   Yes, the information from our yaml file is visible in the output --
   the metadata (name, labels), spec (containers, command, ports) all
   appear. However, Kubernetes ADDED many fields when the pod was deployed:

   - metadata.uid: a unique identifier assigned by Kubernetes
   - metadata.creationTimestamp: when the pod was created
   - metadata.resourceVersion: for concurrency control
   - spec.nodeName: which node the pod was scheduled on
   - spec.serviceAccountName: "default" service account
   - spec.tolerations: default tolerations added by admission controller
   - status: entire status section (phase, conditions, podIP, hostIP,
     containerStatuses with containerID, state, ready flag, etc.)

   Did step 17 create a deployment? No. Creating a Pod directly with
   kubectl create -f pod.yaml creates a standalone Pod, NOT a Deployment.

   $ kubectl get deployments
   NAME    READY   UP-TO-DATE   AVAILABLE   AGE
   nginx   3/3     3            3           6m

   Only the nginx deployment exists. The core-k8s pod is unmanaged --
   if it dies, no controller will recreate it (unlike the nginx pods
   which are managed by the Deployment/ReplicaSet controller).


19. JSONpath queries:

   a. $ kubectl get pods -o=jsonpath='{.items[0].status.phase}'
      Running

      This returns the phase of the first pod (items[0]). The phase
      "Running" means the pod has been bound to a node and all containers
      have been started.

   b. $ kubectl get pods -o=jsonpath='{.items[0].status.podIP}'
      10.244.0.5

      This is the IP address assigned to the first pod from the pod CIDR
      range (10.244.0.0/24). Each pod gets its own unique IP.

   c. $ kubectl get pods -o=jsonpath='{.items[0].status.hostIP}'
      172.18.0.2

      This is the IP of the node hosting the pod. In Kind, the node is a
      Docker container with its own IP on the Docker bridge network.

   Yes, this same information was in the full yaml output from step 18
   under the status section. The jsonpath queries extract specific fields
   from the same data structure.


20. Pod mounts and resolv.conf:

   $ kubectl exec -t -i core-k8s -- mount | grep resolv
   /dev/sda1 on /etc/resolv.conf type ext4 (rw,relatime)

   The pod's resolv.conf is bind-mounted from the node's filesystem.
   Kubernetes creates a custom resolv.conf for each pod and mounts it in.

   $ kubectl exec -t -i core-k8s -- cat /etc/resolv.conf
   nameserver 10.96.0.10
   search default.svc.cluster.local svc.cluster.local cluster.local
   options ndots:5

   The pod's nameserver is 10.96.0.10, which is the ClusterIP of the
   kube-dns (CoreDNS) service.

   $ docker exec -t -i a1b2c3d4e5f6 cat /etc/resolv.conf
   nameserver 192.168.65.254

   The Node's nameserver is 192.168.65.254 (the host's DNS, typically
   provided by Docker Desktop).

   These are DIFFERENT. The pod uses the Kubernetes internal DNS service
   (CoreDNS at 10.96.0.10) for service discovery within the cluster. The
   node uses the external DNS provided by the host/Docker network. This
   separation allows pods to resolve Kubernetes service names (like
   "my-service.default.svc.cluster.local") while the node resolves
   external hostnames through the host's DNS.


21. Output of ps in the pod:

   $ kubectl exec -t -i core-k8s -- ps
   PID   USER     TIME  COMMAND
       1 root      0:00 sleep 10000
      12 root      0:00 ps

   The pod can only see 2 processes: the sleep command (PID 1) and the
   ps command itself. This is because the pod runs in its own PID
   namespace -- it has an isolated view of the process table. The sleep
   command appears as PID 1 inside the pod, even though it has a
   completely different PID on the node. This PID namespace isolation
   is a core container feature that prevents containers from seeing or
   interfering with processes outside their namespace.
