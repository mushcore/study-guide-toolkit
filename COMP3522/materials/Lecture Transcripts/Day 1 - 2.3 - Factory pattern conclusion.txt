 Let's look at some of the benefits of the factory pattern. First is we're separating out the creation logic from the use logic. Second is every time we want to create a new object, all we have to do is create a new subclass for that object that we want to create. Third is we're following a ton of the solid principles, single responsibility, because each factory is responsible for purely creation, open-close, because whenever you want to create a new type of object, all we have to do is subclass each factory, a list-cars substitution, because we have that one magic method that can take in any factory, and it can create whatever object we pass in, and dependency inversion because we're relying on abstractions instead of concretes. And finally, it's great if you don't know ahead of time what kind of objects you want in your system. So for instance, in our previous example, we had members and guests in our form, but maybe in the future this wants to be expanded. So by using the factory pattern, we've kind of already set up our system, so it can support any kind of users that you want. Maybe there's an advanced member or a super guest, and already our system will support that. As for some of the disadvantages, I'm sure by now you've seen that there's a ton of different classes that you have to create for this pattern. So this can make our code more complex and harder to the bug. And second thing is sometimes the classes themselves can be artificial. So let's say you want to create a very minor change to a product, and therefore you have to create a whole new factory just to support that minor change.
