## PART A: MODULE 07 — NETWORK SERVICES (Sendmail, NIS, LDAP, NFS, Samba)

### A.1 SENDMAIL — Mail Transfer Agent (MTA)

#### Key Architecture
- **MTA vs MUA vs MDA**:
  - **MTA** (Mail Transfer Agent): sendmail, postfix, exim — accepts SMTP connections (port 25), routes mail, delivers to MDA
  - **MUA** (Mail User Agent): Outlook, Thunderbird, mutt — client software that composes and reads mail
  - **MDA** (Mail Delivery Agent): procmail, maildrop — delivers to user mailbox after MTA hands off
  - **Sendmail is an MTA**: It accepts incoming SMTP, queues messages, routes, and delivers (or hands to MDA)

#### Sendmail Components
1. **Daemon**: `/usr/sbin/sendmail` — runs as root, listens on port 25
2. **Configuration**: `/etc/mail/sendmail.cf` — complex, generated from `/etc/mail/sendmail.mc`
3. **Aliases**: `/etc/aliases` — maps logical addresses to real users/commands
4. **Mail queue**: `/var/spool/mqueue/` — temporary storage for queued messages
5. **Log**: `/var/log/maillog` — transaction history

#### sendmail.cf vs sendmail.mc
- **sendmail.mc** (macro configuration): Human-readable source file
  - Contains macro definitions, DAEMON_OPTIONS, features
  - Example: `DAEMON_OPTIONS('Port=smtp,Addr=127.0.0.1, Name=MTA')dnl`
  - Modified by sysadmin
- **sendmail.cf** (compiled configuration): Machine-readable output
  - Generated by m4 macro processor: `cd /etc/mail && sudo make`
  - Sendmail reads .cf at startup, not .mc
  - DO NOT edit .cf directly
  - Must restart sendmail after `make`: `sudo systemctl restart sendmail`

#### Critical Configuration: Listening Interface
- **Default (localhost only)**:
  ```
  DAEMON_OPTIONS('Port=smtp,Addr=127.0.0.1, Name=MTA')dnl
  ```
  - Sendmail listens ONLY on 127.0.0.1:25
  - Cannot receive external mail
  - Can still send local and relay mail

- **Modified (all interfaces)**:
  ```
  DAEMON_OPTIONS('Port=smtp, Name=MTA')dnl
  ```
  - Removes `Addr=127.0.0.1`
  - Sendmail listens on 0.0.0.0:25 (all interfaces)
  - CAN receive external mail
  - **Requires firewall rule**: `sudo firewall-cmd --add-service=smtp`

#### Aliases and Newaliases
- **Aliases**: Map logical recipient to actual mailbox or command
  ```
  # /etc/aliases
  root: kevinliang          # root → kevinliang's mailbox
  admin: root               # admin → root
  support: |/usr/bin/ticket # support → pipe to /usr/bin/ticket script
  ```
- **newaliases** command:
  - Reads `/etc/aliases`, generates `/etc/aliases.db`
  - MUST run after editing /etc/aliases (or sendmail reads stale data)
  - Sendmail does NOT auto-detect /etc/aliases changes
  - Usage: `sudo newaliases`

#### Mail Queue Management
- **mailq** (= `sendmail -bp`): Display queue
  ```
  $ mailq
  /var/spool/mqueue is empty
  Total requests: 0
  ```
  - Shows queued messages, recipient, size, date
  - Empty = all delivered

- **Forcing delivery**: `sudo sendmail -q` (process queue now, don't wait)
- **Removing stuck message**: Remove file in `/var/spool/mqueue/`

#### Testing Sendmail
1. **Send local mail** (MUA):
   ```
   $ echo "Test message" | mail -s "Subject" user@localhost
   ```
   - Goes to `/var/spool/mail/user`

2. **Send external mail** (if DAEMON_OPTIONS allows):
   ```
   $ sendmail external@example.com << EOF
   Subject: Test
   From: root@myhost
   Body text
   EOF
   ```

3. **Monitor log**:
   ```
   $ sudo tail -f /var/log/maillog
   ... sendmail[PID]: ... to=<external@example.com>, stat=Sent
   ... sendmail[PID]: ... to=<external@example.com>, stat=Deferred (connection refused)
   ```

#### Lab 7 Example: Sendmail Setup
```bash
# Step 1: Install
$ sudo dnf install sendmail sendmail-cf mailx

# Step 2: Enable and start
$ sudo systemctl enable sendmail
$ sudo systemctl start sendmail

# Step 3: Open firewall for SMTP
$ sudo firewall-cmd --add-service=smtp
$ sudo firewall-cmd --permanent --add-service=smtp

# Step 4: Configure to receive external mail
$ sudo vi /etc/mail/sendmail.mc
# Find: DAEMON_OPTIONS('Port=smtp,Addr=127.0.0.1, Name=MTA')dnl
# Change to: DAEMON_OPTIONS('Port=smtp, Name=MTA')dnl

# Step 5: Rebuild .cf from .mc
$ cd /etc/mail
$ sudo make

# Step 6: Ensure /etc/mail/local-host-names contains domain
$ sudo echo "L327S06A.infosec.bcit.ca" >> /etc/mail/local-host-names

# Step 7: Restart sendmail
$ sudo systemctl restart sendmail

# Step 8: Verify listening on port 25 on all interfaces
$ ss -tlnp | grep :25
LISTEN  0  128  0.0.0.0:25
```

### A.2 NIS — Network Information Service

#### What NIS Does
- **Centralized user/password/host database** over LAN
- **Like LDAP but simpler**: Flat maps (key-value), not hierarchical
- **Common on legacy systems** (UNIX, Linux, some Windows)
- **NOT LDAP**: Different protocol, different architecture
- **NOT DNS**: NIS domain is separate from DNS domain

#### NIS Architecture
1. **NIS Master Server**: Authoritative copy, generates maps, responds to queries
2. **NIS Slave Server**: Read-only copies, loaded from master, handles queries
3. **NIS Client**: Uses ypbind to find server, queries ypserv, caches results
4. **Daemons**:
   - **ypserv**: NIS server (listens on port 111 + dynamic RPC)
   - **ypbind**: NIS client, binds to server
   - **ypxfrd**: Transfer maps from master to slave (port 600)
   - **rpcbind** (portmap): RPC registry (port 111)

#### Maps (Databases)
- **passwd.byname**: Maps username → UID
- **passwd.byuid**: Maps UID → username
- **group.byname**: Maps groupname → GID
- **group.bygid**: Maps GID → groupname
- **hosts.byname**: Maps hostname → IP
- **hosts.byaddr**: Maps IP → hostname
- **services.byname**: Maps service name → port
- **netgroup**: Network-wide user groups

#### NIS Domain
- **NOT the same as DNS domain**
  - DNS: infosec.bcit.ca (resolves names to IPs)
  - NIS: infosec (different namespace, separate from DNS)
- **Set with**: `nisdomainname` or `/etc/sysconfig/network`
  ```
  # /etc/sysconfig/network
  NISDOMAIN=infosec
  ```
- **Verify**: `nisdomainname` or `domainname` (deprecated)

#### Setting Up NIS Master Server
1. **Install NIS packages**:
   ```bash
   $ sudo dnf install nis nis-tools
   ```

2. **Initialize master** (creates maps from /etc/passwd, /etc/group, /etc/hosts):
   ```bash
   $ sudo /usr/lib64/yp/ypinit -m
   ```
   - Prompts for slave servers (enter to skip)
   - Creates `/var/yp/NISDOMAIN/` with map files
   - Generates dbm (key-value) files from source files

3. **Configure /var/yp/securenets** (access control):
   ```
   # Allow only local subnet
   255.255.255.0 192.168.1.0
   ```
   - Master checks this before responding to queries
   - Prevents external NIS leaks

4. **Start daemons**:
   ```bash
   $ sudo systemctl enable ypserv
   $ sudo systemctl start ypserv
   ```

#### Setting Up NIS Client
1. **Install NIS client tools**:
   ```bash
   $ sudo dnf install nis nis-tools
   ```

2. **Set NIS domain**:
   ```bash
   $ sudo nisdomainname infosec
   $ sudo echo "NISDOMAIN=infosec" >> /etc/sysconfig/network
   ```

3. **Edit /etc/nsswitch.conf** (name service switch — where to look for user/group info):
   ```
   # Use local files first, then NIS
   passwd: files nis
   group:  files nis
   hosts:  files nis
   ```
   - Order matters: files checked before nis
   - Fallback if NIS server down: uses local /etc/passwd

4. **Start ypbind**:
   ```bash
   $ sudo systemctl enable ypbind
   $ sudo systemctl start ypbind
   ```

#### NIS Commands
- **ypcat** — Display entire map:
  ```bash
  $ ypcat passwd.byname        # All NIS users
  $ ypcat group.byname         # All NIS groups
  $ ypcat hosts.byname         # All NIS hosts
  ```

- **ypmatch** — Look up single entry:
  ```bash
  $ ypmatch root passwd.byname    # Find root in NIS
  $ ypmatch 127.0.0.1 hosts.byaddr # Find hostname for IP
  ```

- **ypwhich** — Show bound NIS server:
  ```bash
  $ ypwhich
  nismastername
  ```

- **yppasswd** — Change NIS password (updates master):
  ```bash
  $ yppasswd
  (prompts for old, new password)
  ```

#### NIS vs LDAP (Exam Trap)
| Feature | NIS | LDAP |
|---------|-----|------|
| Hierarchy | Flat maps | Tree structure (DN) |
| Domain | NIS domain (separate from DNS) | LDAP suffix (usually matches DNS) |
| Port | 111 (RPC) + dynamic | 389 (LDAP), 636 (LDAPS) |
| Query language | ypcat/ypmatch | ldapsearch with filters |
| Encryption | None (plaintext) | LDAPS (TLS), StartTLS |
| Typical use | Legacy UNIX | Modern Linux/enterprise |

### A.3 LDAP — Lightweight Directory Access Protocol

#### LDAP Architecture
- **slapd daemon**: LDAP server, listens on port 389 (or 636 for LDAPS)
- **Database**: Berkeley DB backend stores directory tree
- **Configuration**: `/etc/openldap/slapd.conf` (older) or `/etc/openldap/slapd.d/` (newer)
- **Client tools**: ldapsearch, ldapadd, ldapmodify, ldapdelete

#### LDAP Schema: DN, RDN, DC, CN, OU, O
- **DN (Distinguished Name)**: Full path to entry, like file system path
  ```
  cn=kevinliang,ou=students,dc=infosec,dc=bcit,dc=ca
  ```
  - Uniquely identifies entry in directory tree

- **RDN (Relative Distinguished Name)**: Single component of DN
  ```
  cn=kevinliang (RDN)
  ou=students (RDN)
  ```

- **DN Components**:
  - **cn=** (commonName): Person's name (unique within parent)
  - **ou=** (organizationalUnit): Department/group
  - **dc=** (domainComponent): Domain level (dc=infosec, dc=bcit, dc=ca)
  - **o=** (organization): Company name
  - **c=** (country): 2-letter country code

- **Example DN**:
  ```
  cn=kevinliang,ou=staff,o=BCIT,c=CA
  ```
  Reads as: kevinliang in staff department of BCIT organization in Canada

#### slapd.conf (Server Configuration)
```
# Root of directory tree
suffix "dc=infosec,dc=bcit,dc=ca"

# Admin user for this suffix
rootdn "cn=admin,dc=infosec,dc=bcit,dc=ca"

# Admin password (hashed by slappasswd)
rootpw {SSHA}W2K...encrypted_hash...

# Database backend
database bdb
directory /var/lib/ldap

# Indexes for fast searches
index objectClass eq
index cn pres,sub,eq
index uid pres,sub,eq
```

#### Key LDAP Attributes
- **objectClass**: Type of entry (person, organizationalUnit, domain, etc.)
- **cn**: Common name (person or resource name)
- **sn**: Surname
- **uid**: User ID (username)
- **userPassword**: Hashed password
- **mail**: Email address
- **telephoneNumber**: Phone
- **ou**: Organizational unit name
- **dc**: Domain component

#### LDIF Format (LDAP Data Interchange Format)
```
dn: cn=kevinliang,ou=students,dc=infosec,dc=bcit,dc=ca
objectClass: inetOrgPerson
objectClass: organizationalPerson
objectClass: person
cn: kevinliang
sn: Liang
uid: kliang
userPassword: {SHA}W2K...
mail: kevinliang@my.bcit.ca
telephoneNumber: +1-604-555-1234
```

#### LDAP Operations
1. **ldapsearch** — Query directory:
   ```bash
   $ ldapsearch -H ldap://ldapserver:389 -x -b "dc=infosec,dc=bcit,dc=ca" "uid=kevinliang"
   # Returns: cn, mail, telephoneNumber, etc.
   ```
   - `-H`: URI (ldap:// or ldaps://)
   - `-x`: Simple auth (no SASL)
   - `-b`: Base DN for search
   - Filter: uid=kevinliang (LDAP filter syntax)

2. **ldapadd** — Add entry:
   ```bash
   $ ldapadd -H ldap://server -D "cn=admin,dc=..." -W -f entry.ldif
   # -D: Bind as admin DN
   # -W: Prompt for password
   # -f: Read from LDIF file
   ```

3. **ldapmodify** — Modify entry:
   ```bash
   $ ldapmodify -H ldap://server -D "cn=admin,dc=..." -W -f changes.ldif
   ```

4. **ldapdelete** — Delete entry:
   ```bash
   $ ldapdelete -H ldap://server -D "cn=admin,dc=..." -W "cn=user,ou=students,dc=..."
   ```

#### Ports
- **389/TCP**: LDAP (unencrypted or STARTTLS)
- **636/TCP**: LDAPS (TLS from start of connection)

### A.4 NFS — Network File System

#### NFS Purpose
- **Mount remote filesystems** over network
- **Transparent to applications**: Access `/mnt/nfs/` like local drive
- **Linux-only**: Samba is for Windows interop
- **Not encrypted** (use SSH tunneling for security)

#### NFS Daemons and Ports
- **nfsd**: NFS server daemon, listens on port 2049 (TCP/UDP)
- **rpcbind** (portmap): Maps RPC service names to ports, port 111
- **mountd**: Mount daemon, handles mount requests
- **statd**: Status monitor, tracks client mounts

#### /etc/exports (Server Configuration)
- **Purpose**: Defines which directories exported (shared) and to whom
- **Syntax**:
  ```
  /exported/path      client-spec(options)
  ```

- **Examples**:
  ```
  # Export /home to all, read-only
  /home               *(ro)
  
  # Export /var/nfs to 192.168.1.0/24, read-write, squash root
  /var/nfs            192.168.1.0/24(rw,root_squash)
  
  # Export to single host with sync
  /data               server2.example.com(rw,sync)
  ```

- **Common Options**:
  - **ro**: Read-only
  - **rw**: Read-write
  - **root_squash**: Map remote root to nobody (prevents root on client from being root on server)
  - **no_root_squash**: Remote root stays root (security risk)
  - **all_squash**: Map all remote users to nobody
  - **sync**: Write to disk before returning (safe, slower)
  - **async**: Return before disk write (faster, risky)
  - **subtree_check**: Check if file in exported subtree (default, slower)
  - **no_subtree_check**: Don't check (default in newer kernels, faster)

#### exportfs Command
- **Update exports without restarting NFS**:
  ```bash
  $ sudo exportfs -a        # Export all from /etc/exports
  $ sudo exportfs -r        # Re-export (refresh)
  $ sudo exportfs -u /path  # Unexport specific path
  ```
- **Show exported**:
  ```bash
  $ sudo exportfs -v
  /var/nfs        192.168.1.0/24(sync,wdelay,root_squash,...)
  ```

#### NFS Client: Manual Mount
```bash
# Mount NFS share
$ sudo mount -t nfs server.example.com:/var/nfs /mnt/nfs

# Options
$ sudo mount -t nfs -o rw,hard,intr server:/share /mnt

# Verify
$ mount | grep nfs
server.example.com:/var/nfs on /mnt/nfs type nfs (...)

# Unmount
$ sudo umount /mnt/nfs
```

#### NFS Client: Automounting with autofs
- **autofs**: Automatically mount/unmount on demand
- **Daemons**: autofs, automountd
- **Configuration**:
  1. **Master map** (`/etc/auto.master`):
     ```
     /-           /etc/auto.direct      # Direct mounts
     /mnt/nfs     /etc/auto.nfs         # Indirect mounts
     ```

  2. **Indirect mount map** (`/etc/auto.nfs`):
     ```
     # Key: mount point relative to /mnt/nfs
     home    -rw,hard,intr    server:/home
     share   -ro              server:/var/nfs
     ```
     - When accessed `/mnt/nfs/home`, autofs mounts server:/home
     - After 10 minutes inactivity, unmounts automatically

  3. **Direct mount map** (`/etc/auto.direct`):
     ```
     /nfs/server1/home    -rw    server1:/home
     /nfs/server2/data    -rw    server2:/data
     ```
     - Defines exact mount paths

- **Enable**:
  ```bash
  $ sudo systemctl enable autofs
  $ sudo systemctl start autofs
  ```

- NFS is mentioned but not deeply tested in lab
- Lab 3 focuses on DHCP server setup (pool assignment, fixed hosts)

#### NFS vs Samba (Exam Trap)
- **NFS**: Linux-to-Linux file sharing, POSIX permissions, fast
- **Samba**: Windows interop, SMB protocol, NTFS permissions, slower

### A.5 SAMBA — Windows File/Printer Sharing

#### Samba Purpose
- **Enable Linux to share files/printers with Windows**
- **Windows sees Linux as another Windows machine**
- **Also supports Linux-to-Linux** (but NFS is better)
- **Protocol**: SMB (Server Message Block) / CIFS (Common Internet File System)

#### Samba Daemons and Ports
- **smbd**: Main daemon, handles file/printer sharing, port 139 (NetBIOS) + 445 (CIFS)
- **nmbd**: NetBIOS name server, port 137 (UDP) + 138 (UDP)
- **winbindd**: Windows domain integration (advanced)
- **swat**: Web-based config tool on port 901 (DEPRECATED, overwrites smb.conf)

#### smb.conf (Main Configuration)
- **Location**: `/etc/samba/smb.conf`
- **Sections**: [global], [homes], [printers], [sharename], ...
- **Syntax**: Key = value, comments with `#` or `;`

- **[global] Section** (server-wide settings):
  ```
  [global]
     workgroup = WORKGROUP           # Windows workgroup name
     server string = Lab 7 Samba Server
     netbios name = SAMBA_SERVER     # Name Windows sees
     security = user                 # Auth method (user, domain, share)
     passdb backend = tdbsam         # Password DB (tdbsam or ldapsam)
     map to guest = Bad User         # Guest access for bad passwords
     log file = /var/log/samba/log.%m    # Per-machine log
     max log size = 1000             # KB before rotate
  ```

- **[homes] Section** (user home directories):
  ```
  [homes]
     comment = Home Directories
     browseable = no                 # Don't show in browse list
     read only = no                  # Allow write
     valid users = %S                # Only owner can access
  ```

- **[printers] Section**:
  ```
  [printers]
     comment = All Printers
     path = /var/spool/samba
     printer admin = @lpadmin
  ```

- **[sharename] Section** (custom share):
  ```
  [shared]
     comment = Lab 7 Shared Directory
     path = /srv/samba/shared        # Where on disk
     browseable = yes                # Show in browse list
     read only = no                  # Allow write
     valid users = fred, @staff      # Who can access
     create mask = 0664              # New file permissions
     directory mask = 0775           # New dir permissions
     guest ok = no                   # Require auth
  ```

#### Samba User Management
1. **Create Linux user**:
   ```bash
   $ sudo useradd fred
   ```

2. **Add to Samba (creates Samba password)**:
   ```bash
   $ sudo smbpasswd -a fred
   New SMB password: fredpass
   Retype new SMB password: fredpass
   Added user fred.
   ```
   - Stores in `/etc/samba/private/passdb.tdb`
   - NOT the same as Linux password
   - Can use `-d` to disable, `-e` to enable

3. **Verify configuration**:
   ```bash
   $ testparm
   Load smb config files from /etc/samba/smb.conf
   Loaded services file OK.
   Server role: ROLE_STANDALONE
   Press enter to see a dump of your configuration...
   ```
   - No output = syntax OK
   - If errors, testparm shows them

#### Samba Client: Linux
- **smbclient** (command-line FTP-like interface):
  ```bash
  $ smbclient //localhost/shared -U fred
  Enter WORKGROUP\fred's password: fredpass
  smb: \> ls                  # List shares
  smb: \> get file.txt -      # Download to stdout
  smb: \> put local.txt       # Upload
  smb: \> quit
  ```

- **mount.cifs** (mount as filesystem):
  ```bash
  $ sudo mkdir /mnt/samba
  $ sudo mount -t cifs //server/shared /mnt/samba -o username=fred,password=fredpass
  $ ls /mnt/samba
  $ sudo umount /mnt/samba
  ```

#### Samba Client: Windows
1. Open File Explorer
2. Type in address bar:
   ```
   \\server.example.com\shared
   ```
3. Enter credentials when prompted:
   - Username: fred
   - Password: fredpass

Or command-line:
```cmd
> net use Z: \\server\shared /user:fred fredpass
> dir Z:\
```

#### Security Modes
- **security = user**: Credentials validated against passdb.tdb (most common)
- **security = share**: Share-level password (old, not recommended)
- **security = domain**: Authenticate against Windows domain controller
- **security = ads**: Active Directory integration

#### SWAT Warning (Exam Trap)
- **swat** (port 901): Web-based config tool
- **DANGER**: `swat` rewrites entire `/etc/samba/smb.conf`
  - Can overwrite custom settings
  - Not recommended for production
- **If used**: Always backup smb.conf first
  ```bash
  $ sudo cp /etc/samba/smb.conf /etc/samba/smb.conf.backup
  ```

## PART B: MODULE 08 — DNS, iptables, APACHE

### B.1 DNS — BIND (Berkeley Internet Name Domain)

#### DNS Purpose
- **Resolve domain names to IP addresses** (DNS A record)
- **Inverse: IP to name** (reverse DNS, PTR record)
- **Distribute web/mail/other services** (MX, SRV, CNAME records)

#### BIND Daemons and Ports
- **named**: Main DNS daemon
- **Port 53/TCP**: Zone transfers, large queries
- **Port 53/UDP**: Standard DNS queries (fast, small packets)
- **Port 953/TCP**: rndc (remote name daemon control, admin only)

#### named.conf (BIND Configuration)
- **Location**: `/etc/named.conf`
- **Structure**:
  ```
  options {
      listen-on port 53 { 127.0.0.1; 192.168.1.1; };
      directory "/var/named";
      allow-query { localhost; 192.168.1.0/24; };
      recursion yes;
  };
  
  logging {
      channel default_debug {
          file "data/named.run";
          severity dynamic;
      };
  };
  
  zone "example.com" {
      type master;
      file "/etc/named/zones/db.example.com";
  };
  
  zone "1.168.192.in-addr.arpa" {
      type master;
      file "/etc/named/zones/db.192.168.1";
  };
  ```

- **Key directives**:
  - **listen-on**: IP addresses BIND listens on
  - **directory**: Where zone files located
  - **allow-query**: Who can query this server
  - **recursion**: Whether to resolve queries recursively (caching server) or just answer authoritatively
  - **zone**: Define zone (domain to serve)

#### Zone Types
- **master**: Authoritative zone, reads from zone file
- **slave**: Secondary zone, transfers from master, serves read-only
- **forward**: Forward queries to another server
- **hint**: Root zone (.)

#### Zone Files (db.example.com)
```
$TTL    604800
@       IN      SOA     ns1.example.com. admin.example.com. (
                    1     ; Serial
               604800     ; Refresh (slaves recheck in 604800s)
                86400     ; Retry (if transfer fails)
              2419200     ; Expire (discard if no sync)
               604800 )   ; Negative Cache TTL

        IN      NS      ns1.example.com.   ; Nameserver
        IN      NS      ns2.example.com.   ; Secondary NS

ns1     IN      A       192.168.1.1        ; A record
ns2     IN      A       192.168.1.2        ; A record
www     IN      A       192.168.1.10       ; www.example.com → 192.168.1.10
mail    IN      A       192.168.1.11       ; mail.example.com
        IN      MX  10  mail               ; Mail server priority 10

ftp     IN      CNAME   www                ; ftp.example.com → www.example.com

_http   IN      SRV  10 60 80 www          ; Service record
```

- **SOA (Start of Authority)**:
  - Serial: Version number, increment when zone changes, slaves use to detect updates
  - Refresh: How often slaves check for updates
  - Retry: Wait before retrying failed transfer
  - Expire: Discard zone if transfer fails for this long
  - Negative TTL: Cache non-existent records for this long

- **Record Types**:
  - **A**: IPv4 address (most common)
  - **AAAA**: IPv6 address
  - **CNAME**: Alias (www → main server)
  - **MX**: Mail exchange (priority field)
  - **NS**: Nameserver for zone
  - **PTR**: Reverse DNS (IP to name)
  - **TXT**: Text (SPF, DKIM, etc.)
  - **SRV**: Service location

#### Reverse Zone (db.192.168.1)
- **Purpose**: IP address → hostname lookups (PTR records)
- **Zone name**: `1.168.192.in-addr.arpa` (reversed octets)
- **Content**:
  ```
  $TTL 604800
  @    IN SOA ns1.example.com. admin.example.com. (...)
       IN NS  ns1.example.com.
  1    IN PTR ns1.example.com.       ; 192.168.1.1 → ns1
  10   IN PTR www.example.com.       ; 192.168.1.10 → www
  11   IN PTR mail.example.com.      ; 192.168.1.11 → mail
  ```

### B.2 iptables — Packet Filtering and Network Address Translation

#### iptables Architecture: Tables and Chains

**5 Chains in filter table** (built-in):
- **INPUT**: Packets destined for local machine
- **OUTPUT**: Packets originating from local machine
- **FORWARD**: Packets passing through (router)
- **PREROUTING**: Packets just arriving (before routing decision)
- **POSTROUTING**: Packets just leaving (after routing decision)

**4 Tables** (different purposes):
1. **filter** (default): INPUT, OUTPUT, FORWARD — controls packet acceptance
2. **nat** (network address translation): PREROUTING, POSTROUTING, OUTPUT — modifies source/dest addresses
3. **mangle**: PREROUTING, POSTROUTING, INPUT, OUTPUT, FORWARD — modifies packet headers (ToS, TTL, etc.)
4. **raw**: PREROUTING, OUTPUT — connection tracking bypass (advanced)

#### Packet Flow Diagram
```
Incoming packet
    ↓
[PREROUTING] (nat) — DNAT, REDIRECT
    ↓
Routing decision
    ↓
        ↙               ↘
    Local?              Forwarding?
        ↓                   ↓
    [INPUT] (filter)    [FORWARD] (filter)
        ↓                   ↓
    Application         [POSTROUTING] (nat) — SNAT, MASQUERADE
        ↓                   ↓
    [OUTPUT] (filter)    Out interface
        ↓
        └───────────────┘
            ↓
    [POSTROUTING] (nat)
            ↓
        Out interface
```

#### iptables Rules
- **Format**:
  ```
  iptables -A CHAIN -p PROTOCOL -s SOURCE -d DEST --dport PORT -j TARGET
  ```

- **Common options**:
  - `-A`: Append rule to chain
  - `-I`: Insert rule at position
  - `-D`: Delete rule
  - `-F`: Flush all rules in chain
  - `-L`: List rules
  - `-p`: Protocol (tcp, udp, icmp, all)
  - `-s`: Source IP (with optional /mask)
  - `-d`: Destination IP
  - `-i`: Incoming interface (eth0)
  - `-o`: Outgoing interface
  - `--sport`: Source port
  - `--dport`: Destination port
  - `-j`: Jump to target (ACCEPT, DROP, REJECT, LOG, etc.)

#### Targets
- **ACCEPT**: Allow packet through
- **DROP**: Discard packet silently (no response)
- **REJECT**: Discard and send error response (ICMP port unreachable)
- **LOG**: Log packet to syslog (doesn't stop it)
- **SNAT**: Source NAT — change source IP (for outgoing)
- **DNAT**: Destination NAT — change destination IP (for incoming)
- **MASQUERADE**: Like SNAT but auto-detects source IP (for DHCP interfaces)
- **REDIRECT**: Redirect to local port

#### Common Rules
```bash
# Accept SSH from subnet
iptables -A INPUT -p tcp -s 192.168.1.0/24 --dport 22 -j ACCEPT

# Block all incoming to port 80 except from 10.0.0.0/8
iptables -A INPUT -p tcp --dport 80 -j DROP
iptables -I INPUT -p tcp -s 10.0.0.0/8 --dport 80 -j ACCEPT

# Log dropped packets
iptables -A INPUT -j LOG --log-prefix "DROP: "
iptables -A INPUT -j DROP

# DNAT: Redirect incoming port 8080 to 192.168.1.10:80
iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to 192.168.1.10:80

# SNAT: Change source IP of outgoing packets
iptables -t nat -A POSTROUTING -o eth0 -j SNAT --to 192.168.1.1

# MASQUERADE: NAT for DHCP interface (source IP auto-detected)
iptables -t nat -A POSTROUTING -o ppp0 -j MASQUERADE

# Enable port forwarding (router mode)
echo 1 > /proc/sys/net/ipv4/ip_forward
```

#### Default Policy (Exam Trap)
- **Default policy**: Action taken if NO RULES MATCH
  ```bash
  iptables -P INPUT DROP        # Default to drop incoming
  iptables -P OUTPUT ACCEPT     # Default to accept outgoing
  iptables -P FORWARD DROP      # Default to drop forwarding
  ```
- **If no rule matches, default policy is applied**
  - If policy is DROP: packet discarded
  - If policy is ACCEPT: packet allowed
  - Common mistake: Set default to DROP without ACCEPT rules first → lock yourself out

#### Persisting iptables Rules
- **Default**: Rules lost on reboot
- **Methods**:
  1. **iptables-save/iptables-restore** (older):
     ```bash
     $ sudo iptables-save > /etc/sysconfig/iptables
     $ sudo systemctl enable iptables
     ```
  2. **firewalld** (newer, recommended):
     ```bash
     $ sudo systemctl enable firewalld
     $ sudo firewall-cmd --permanent --add-port=80/tcp
     $ sudo firewall-cmd --reload
     ```

#### firewall-cmd (Modern Alternative to iptables)
- **Wrapper around iptables/nftables**
- **Advantages**: Easier syntax, zone-based, hot reload
- **Common commands**:
  ```bash
  sudo firewall-cmd --list-all              # Show all rules
  sudo firewall-cmd --add-service=http      # Add HTTP (temp)
  sudo firewall-cmd --permanent --add-service=http  # Permanent
  sudo firewall-cmd --add-port=8080/tcp     # Add port (temp)
  sudo firewall-cmd --permanent --add-port=8080/tcp # Permanent
  sudo firewall-cmd --reload                # Apply changes
  ```

- Primarily in slides
- Key exam concepts: chain identification, table identification, packet flow, rule syntax

### B.3 APACHE — HTTP Web Server

#### Apache Daemons and Ports
- **httpd**: Main daemon (runs as root, child processes as apache user)
- **Port 80**: HTTP (unencrypted)
- **Port 443**: HTTPS (with mod_ssl)

#### httpd.conf Structure
- **Location**: `/etc/httpd/conf/httpd.conf`
- **Three main sections**:
  1. **Global Configuration**: Default settings for all virtual hosts
  2. **Main Server Configuration**: Settings for main server (before VirtualHost)
  3. **Virtual Hosts**: Separate configurations per domain

#### Global Settings (Partial)
```
# Default user/group
User apache
Group apache

# Server root (default for relative paths)
ServerRoot /etc/httpd

# Include other config files
Include conf.modules.d/*.conf
Include conf.d/*.conf
```

#### Main Server Section
```
# Default binding
Listen 80

# Document root (where HTML files)
DocumentRoot /var/www/html

# Default directory permissions
<Directory "/var/www/html">
    AllowOverride AuthConfig      # Allow .htaccess overrides
    Require all granted            # Allow all access
</Directory>

# Log files
ErrorLog logs/error_log
CustomLog logs/access_log combined

# Default files to serve
DirectoryIndex index.html index.php

# Module configuration
LoadModule ssl_module modules/mod_ssl.so
```

#### Directory Blocks
- **Purpose**: Set permissions for specific directories
- **Syntax**:
  ```
  <Directory "/var/www/html">
      Require all granted        # Allow access
      Options Indexes FollowSymLinks  # What to allow
      AllowOverride None          # No .htaccess
  </Directory>
  ```

- **Require directives**:
  - `Require all granted`: Allow all
  - `Require all denied`: Deny all
  - `Require ip 192.168.1.0/24`: Allow specific IPs
  - `Require valid-user`: Require authentication

- **Options**:
  - `Indexes`: Show directory listing if no index file
  - `FollowSymLinks`: Follow symbolic links
  - `MultiViews`: Content negotiation (serve index.html.en, index.html.fr)
  - `IncludesNoExec`: Server-side includes without exec
  - `ExecCGI`: Allow CGI execution

#### .htaccess (Per-directory Configuration)
- **Purpose**: Override httpd.conf settings for specific directory
- **Must enable**: `AllowOverride AuthConfig` (or All)
- **Located in**: Directory on disk (e.g., `/var/www/html/protected/.htaccess`)

- **Common settings**:
  ```
  # Require authentication
  AuthType Basic
  AuthName "Restricted Area"
  AuthUserFile /etc/httpd/.htpasswd
  Require valid-user
  
  # Rewrite rules
  RewriteEngine On
  RewriteRule ^index\.php$ - [L]
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteCond %{REQUEST_FILENAME} !-d
  RewriteRule . /index.php [L]
  ```

#### htpasswd (User Authentication)
- **Purpose**: Create password file for HTTP Basic Auth
- **Usage**:
  ```bash
  # Create file with user fred
  $ sudo htpasswd -c /etc/httpd/.htpasswd fred
  New password: fredpass
  Re-type new password: fredpass
  
  # Add another user (without -c, which would recreate file)
  $ sudo htpasswd /etc/httpd/.htpasswd alice
  
  # Verify
  $ sudo cat /etc/httpd/.htpasswd
  fred:$apr1$...hash...
  alice:$apr1$...hash...
  ```

- **In .htaccess**:
  ```
  AuthUserFile /etc/httpd/.htpasswd
  AuthType Basic
  AuthName "My Site"
  Require valid-user
  ```

#### Virtual Hosts
- **Purpose**: Host multiple domains on one server
- **Syntax**:
  ```
  <VirtualHost *:80>
      ServerName example.com
      ServerAlias www.example.com
      DocumentRoot /var/www/example.com/html
      ErrorLog logs/example.com-error_log
      CustomLog logs/example.com-access_log combined
  </VirtualHost>
  
  <VirtualHost *:443>
      ServerName secure.example.com
      DocumentRoot /var/www/secure.com/html
      SSLEngine on
      SSLCertificateFile /etc/ssl/certs/server.crt
      SSLCertificateKeyFile /etc/ssl/private/server.key
  </VirtualHost>
  ```

- **VirtualHost ports**:
  - `*:80`: Listen on all IPs, port 80
  - `192.168.1.1:80`: Listen on specific IP
  - `*:443`: HTTPS

#### User Home Directories (~user)
- **Enable in httpd.conf**:
  ```
  LoadModule userdir_module modules/mod_userdir.so
  
  <Directory "/home/*/public_html">
      AllowOverride FileInfo AuthConfig
      Require all granted
  </Directory>
  ```

- **Or edit `/etc/httpd/conf.d/userdir.conf`**:
  ```
  UserDir public_html  # Look in ~/public_html/
  ```

- **Access**: `http://server.com/~username/`

- **Lab 8 uses ~/html** (not public_html):
  ```
  UserDir html        # Look in ~/html/
  ```

#### mod_ssl (HTTPS)
- **Install**: `sudo dnf install mod_ssl`
- **Enable**: `LoadModule ssl_module modules/mod_ssl.so` in httpd.conf
- **Configure**:
  ```
  <VirtualHost *:443>
      SSLEngine on
      SSLCertificateFile /etc/pki/tls/certs/server.crt
      SSLCertificateKeyFile /etc/pki/tls/private/server.key
      SSLCertificateChainFile /etc/pki/tls/certs/ca-bundle.crt
  </VirtualHost>
  ```

#### Verification and Testing
- **Check syntax**: `sudo httpd -t` or `sudo apachectl configtest`
  ```

  ```

- **List virtual hosts**: `sudo httpd -S`
  ```

  *:80   example.com (/etc/httpd/conf.d/example.conf:1)
  *:443  secure.example.com (/etc/httpd/conf.d/secure.conf:1)
  ```

- **Restart Apache**:
  ```bash

  ```

- **Check running**:
  ```bash

  $ ss -tlnp | grep 80
  ```

## PART C: EXAM FLASHCARDS (30+)

### Sendmail Flashcards
1. **Q: What is the difference between sendmail.cf and sendmail.mc?**
   A: sendmail.mc is human-readable macro config source; sendmail.cf is machine-readable compiled output generated by m4. Sendmail reads .cf at startup, not .mc.

2. **Q: How do you enable Sendmail to receive external mail?**
   A: Edit sendmail.mc, change `DAEMON_OPTIONS('Port=smtp,Addr=127.0.0.1, Name=MTA')` to `DAEMON_OPTIONS('Port=smtp, Name=MTA')` (remove Addr), then `cd /etc/mail && sudo make` and restart.

3. **Q: What does `sudo newaliases` do?**
   A: Reads /etc/aliases, generates /etc/aliases.db. Must run after editing aliases or sendmail won't see changes.

4. **Q: What is the purpose of /var/spool/mqueue/?**
   A: Temporary storage for queued mail messages waiting to be delivered.

5. **Q: What command shows queued mail?**
   A: `mailq` (or `sendmail -bp`). Empty output means all delivered.

### NIS Flashcards
6. **Q: What is the difference between NIS domain and DNS domain?**
   A: NIS domain (e.g., "infosec") is a flat namespace for user/group databases, separate from DNS domain (e.g., "infosec.bcit.ca"). NIS domain is set by `nisdomainname` or in /etc/sysconfig/network.

7. **Q: What does `ypinit -m` do?**
   A: Initializes NIS master server, creates maps from /etc/passwd, /etc/group, /etc/hosts in /var/yp/NISDOMAIN/.

8. **Q: What does /var/yp/securenets control?**
   A: Access control for NIS master server, specifies which networks can query the server. Prevents external leakage.

9. **Q: What is the purpose of `ypbind`?**
   A: NIS client daemon that binds to NIS server, enabling local user/group lookups via NIS.

10. **Q: What command displays all NIS users?**
    A: `ypcat passwd.byname`

11. **Q: What command looks up a single NIS entry?**
    A: `ypmatch USERNAME passwd.byname` (or `ypmatch IP hosts.byaddr` for reverse lookup)

12. **Q: Which file on an NIS client controls where to look for user info (local vs NIS)?**
    A: `/etc/nsswitch.conf` (name service switch). Example: `passwd: files nis` means check local files first, then NIS.

### LDAP Flashcards
13. **Q: What is a DN in LDAP?**
    A: Distinguished Name, the full path to an LDAP entry. Example: `cn=kevinliang,ou=students,dc=infosec,dc=bcit,dc=ca`

14. **Q: What does "cn=" mean in LDAP?**
    A: commonName — typically a person's name or resource name.

15. **Q: What does "dc=" mean in LDAP?**
    A: domainComponent — part of the domain hierarchy. dc=infosec,dc=bcit,dc=ca represents infosec.bcit.ca

16. **Q: What LDAP daemon provides the directory service?**
    A: `slapd` (standalone LDAP daemon)

17. **Q: What are the standard LDAP ports?**
    A: 389/TCP (unencrypted or STARTTLS), 636/TCP (LDAPS with TLS)

18. **Q: What command adds an entry to LDAP?**
    A: `ldapadd` with LDIF file input. Example: `ldapadd -H ldap://server -D "cn=admin,dc=..." -W -f entry.ldif`

19. **Q: What is LDIF format?**
    A: LDAP Data Interchange Format, text-based representation of LDAP entries (dn:, objectClass:, cn:, etc.)

### NFS Flashcards
20. **Q: What port does NFS use?**
    A: 2049/TCP and 2049/UDP

21. **Q: What is the purpose of /etc/exports?**
    A: Defines which directories are shared via NFS and to whom, with options (ro/rw, root_squash, sync, etc.)

22. **Q: What does `root_squash` do in /etc/exports?**
    A: Maps remote root user to nobody, preventing root on client from being root on NFS server (security).

23. **Q: What command updates /etc/exports without restarting NFS?**
    A: `sudo exportfs -a` (export all) or `exportfs -r` (refresh)

24. **Q: What command shows exported NFS shares?**
    A: `sudo exportfs -v` or `showmount -e localhost`

25. **Q: How do you mount an NFS share?**
    A: `sudo mount -t nfs server:/path /mnt/nfs` or with options `sudo mount -t nfs -o rw,hard,intr server:/path /mnt/nfs`

26. **Q: What is autofs used for in NFS?**
    A: Automatically mounts/unmounts NFS shares on demand, configured in /etc/auto.master and /etc/auto.nfs

### Samba Flashcards
27. **Q: What is Samba used for?**
    A: Enables Linux to share files/printers with Windows systems using SMB/CIFS protocol.

28. **Q: What daemons does Samba use?**
    A: `smbd` (file/printer sharing), `nmbd` (NetBIOS name service)

29. **Q: What does `smbpasswd -a fred` do?**
    A: Adds Samba user fred with a password, stored in /etc/samba/private/passdb.tdb. Note: Different from Linux password.

30. **Q: What command validates smb.conf syntax?**
    A: `testparm` — shows any configuration errors.

31. **Q: What is the SWAT danger (Exam Trap)?**
    A: SWAT (port 901) rewrites entire /etc/samba/smb.conf, overwriting custom settings. Always backup first.

32. **Q: How do you access Samba share from Linux command line?**
    A: `smbclient //server/share -U username` (interactive) or `sudo mount -t cifs //server/share /mnt -o username=...`

33. **Q: How do you test Samba from Windows?**
    A: Open File Explorer, type `\\server.ip\sharename` in address bar, enter credentials.

### DNS Flashcards
34. **Q: What ports does DNS use?**
    A: 53/UDP (standard queries), 53/TCP (zone transfers, large queries)

35. **Q: What does "named" daemon do?**
    A: BIND DNS server daemon, answers DNS queries and serves zones.

36. **Q: What command validates named.conf syntax?**
    A: `sudo named-checkconf` — no output means OK.

37. **Q: What command validates a zone file?**
    A: `sudo named-checkzone domainname /path/to/zonefile`

38. **Q: What is the SOA record?**
    A: Start of Authority record, contains zone administrator contact, serial number, refresh/retry/expire times, negative TTL.

39. **Q: What is a CNAME record?**
    A: Canonical Name record, an alias. Example: `ftp IN CNAME www` makes ftp.example.com point to www.example.com

40. **Q: What is a reverse zone in DNS?**
    A: Maps IP addresses to hostnames (PTR records), zone name like `1.168.192.in-addr.arpa` (reversed octets).

41. **Q: What does `dig www.example.com` show?**
    A: Query response with QUESTION section, ANSWER section (A record), and query time in milliseconds.

### iptables Flashcards
42. **Q: What are the 5 chains in the filter table?**
    A: INPUT (local dest), OUTPUT (local source), FORWARD (passing through), PREROUTING (arriving), POSTROUTING (leaving)

43. **Q: What is the difference between DROP and REJECT?**
    A: DROP silently discards packet; REJECT sends ICMP error response.

44. **Q: What does DNAT do?**
    A: Destination NAT — changes destination IP address of incoming packets. Used in nat table PREROUTING.

45. **Q: What does SNAT do?**
    A: Source NAT — changes source IP address of outgoing packets. Used in nat table POSTROUTING.

46. **Q: What does MASQUERADE do?**
    A: Like SNAT but auto-detects source IP, useful for DHCP interfaces. Used in nat table POSTROUTING.

47. **Q: What is the default policy? (Exam Trap)**
    A: Action taken if NO RULE MATCHES. If policy is DROP, packet discarded. If policy is ACCEPT, packet allowed. Set with `iptables -P CHAIN POLICY`.

    A: `sudo iptables -L` (filter table) or `sudo iptables -t nat -L` (nat table)

    A: `iptables -P INPUT DROP` (default drop), then `iptables -A INPUT -p tcp --dport 22 -j ACCEPT`

### Question 6: DNS Zone File Interpretation (Short Answer)
**Q: In a DNS zone file, what does the following SOA record tell you?**
```
@  IN  SOA  ns1.example.com. admin.example.com. (
            1       ; Serial
       604800       ; Refresh
        86400       ; Retry
      2419200       ; Expire
       604800 )     ; Negative Cache TTL
```

**A:**
- **Serial 1**: Version number of the zone file. Slave servers use this to detect when the zone has been updated.
- **Refresh 604800** (7 days): Slave servers check for zone updates every 604800 seconds.
- **Retry 86400** (1 day): If a transfer fails, retry after 86400 seconds.
- **Expire 2419200** (28 days): If the slave server has not synced for 28 days, discard the zone data.
- **Negative TTL 604800** (7 days): If a name does not exist (NXDOMAIN), cache that response for 7 days.

This configuration is typical for a zone with secondary servers; the refresh interval is long (7 days) indicating infrequent updates.

### Question 7: iptables Packet Flow (Short Answer)
**Q: A packet arrives on eth0 destined for the local machine (port 80, TCP). Which chain(s) in the filter table will the packet traverse, and in what order?**

**A:**
1. **PREROUTING** (nat table, NOT filter table — note the distinction): DNAT rules applied before routing decision
2. **Routing decision**: System determines packet is for local machine
3. **INPUT** (filter table): Main filtering happens here. Rules with `iptables -A INPUT -p tcp --dport 80 -j ACCEPT` are evaluated.
4. If accepted in INPUT, packet is delivered to the local application (httpd listening on port 80)

**Note**: If the default INPUT policy is DROP and no rule allows port 80, the packet is discarded (dropped silently).

### Question 9: iptables Default Policy (Exam Trap - Multiple Choice)
**Q: You set the default INPUT policy to DROP. You then add a rule: `iptables -A INPUT -p tcp --dport 22 -j ACCEPT`. An SSH client tries to connect. What happens?**

A) Connection accepted because we added the ACCEPT rule  
B) Connection accepted because the default policy is DROP  
C) Connection dropped because the rule comes after DROP in the chain  
D) Connection may be accepted, depending on packet order ← CORRECT (More nuanced: Connection is accepted; the ACCEPT rule matches before default policy is applied)

**Explanation**: Rules are evaluated in order. The first matching rule is applied; subsequent rules (including default policy) are not checked. Since the SSH rule ACCEPT comes before the default DROP policy, SSH is accepted.

**Real Answer**: The connection is ACCEPTED because the `-A INPUT -p tcp --dport 22 -j ACCEPT` rule matches (SSH on port 22), and the default policy is only applied if NO RULES MATCH.

### Question 12: LDAP vs NIS (Essay - 10 points)
**Q: Compare and contrast NIS and LDAP. When would you use each, and why?**

**A:**

| Aspect | NIS | LDAP |
|--------|-----|------|
| Structure | Flat key-value maps | Hierarchical tree (DN-based) |
| Domain | NIS domain (separate from DNS) | LDAP suffix (usually matches DNS) |
| Security | No encryption, plaintext | LDAPS (TLS) or StartTLS encryption |
| Scalability | Limited to local network | Scales to large networks and forests |
| Protocol | RPC-based, port 111 | LDAP, port 389/636 |
| Use case | Legacy UNIX systems, small LANs | Modern enterprise, Windows AD integration |

**When to use NIS:**
- Legacy systems that expect NIS
- Small local networks where security is not a concern
- Systems that pre-date LDAP adoption

**When to use LDAP:**
- Modern infrastructure with mixed OS (Linux, Windows, Mac)
- Security-conscious environments requiring encryption
- Large organizations needing a centralized directory
- Integration with Windows Active Directory (via Samba or SSSD)
- Scalability across multiple sites

**Best practice**: Modern systems use LDAP with TLS encryption, or integrate with Active Directory using SSSD (System Security Services Daemon) or Samba.

### Question 13: iptables MASQUERADE (Short Answer)
**Q: On a Linux router with a DHCP-assigned IP on ppp0, how do you enable NAT (masquerading) so internal clients (192.168.1.0/24) can reach the internet?**

**A:**
```bash
# Step 1: Enable IP forwarding
$ echo 1 > /proc/sys/net/ipv4/ip_forward
# Or permanently in /etc/sysctl.conf: net.ipv4.ip_forward = 1

# Step 2: Allow FORWARD chain
$ sudo iptables -A FORWARD -i eth0 -o ppp0 -j ACCEPT
$ sudo iptables -A FORWARD -i ppp0 -o eth0 -j ACCEPT

# Step 3: Add MASQUERADE rule (auto-detect source IP for DHCP)
$ sudo iptables -t nat -A POSTROUTING -o ppp0 -j MASQUERADE

# Step 4: Persist rules
$ sudo iptables-save > /etc/sysconfig/iptables
$ sudo systemctl enable iptables
```

The MASQUERADE target is preferred over SNAT for DHCP interfaces because it automatically detects the interface's IP (which may change).

## PART E: EXAM TRAPS AND GOTCHAS

### Trap 1: NFS vs Samba (Linux-only vs Windows Interop)
**The Trap**: On a midterm or final, a question asks "You have a Linux server that needs to share files with both Linux clients and Windows clients. Which should you use: NFS or Samba?"

**The Wrong Answer**: "NFS" (applicatble only to Linux)

**The Right Answer**: "Samba" (Windows interop) — use Samba for Windows, NFS for Linux-only. If you must support both, use Samba with appropriate permissions.

**Why it matters**: NFS is optimized for Linux (POSIX permissions), Samba is optimized for Windows (SMB/CIFS, NTFS permissions). Trying to use NFS with Windows is clunky; trying to use Samba as primary Linux-to-Linux file share is slower than NFS.

---

### Trap 2: NIS Domain ≠ DNS Domain
**The Trap**: Question: "What is the NIS domain for company.com?"

**The Wrong Answer**: "company.com" (confusing NIS domain with DNS domain)

**The Right Answer**: NIS domain is a separate namespace, often a short name like "company" or "internal", NOT the same as DNS domain. Set with `nisdomainname` or in /etc/sysconfig/network. A company might have:
- DNS domain: company.com
- NIS domain: company
- These are independent

**Why it matters**: NIS and DNS are different services. NIS domain is set locally on each client/server. DNS domain is global and resolves names. Confusing them causes misconfiguration.

---

### Trap 3: iptables Default Policy (If No Rule Matches)
**The Trap**: Question: "You set default INPUT policy to DROP. No rules are defined. A ping arrives. What happens?"

**The Wrong Answer**: "The ping is accepted" (forgetting default policy applies when no rule matches)

**The Right Answer**: The ping is dropped (default policy DROP applies because no rule matches the incoming ping).

**Corollary Trap**: "You add rule `-A INPUT -p tcp --dport 80 -j ACCEPT` but set default policy to DROP. SSH connects. What happens?"
- **Wrong Answer**: "SSH is dropped because default is DROP"
- **Right Answer**: "SSH is dropped because the ACCEPT rule only matches port 80, not port 22. Default DROP applies to SSH."

**Why it matters**: Default policy determines what happens when NO RULE MATCHES. Rules are checked in order; first match wins. If no match, default policy is applied.

---

### Trap 4: Apache DocumentRoot is /var/www/html in RHEL/Fedora
**The Trap**: Question: "Where does Apache serve files from by default on RHEL?"

**The Wrong Answer**: "/var/www/" or "/usr/share/httpd/"

**The Right Answer**: "/var/www/html/" (note the /html subdirectory) on RHEL/Fedora. On Debian/Ubuntu, it's /var/www/html/ as well (same by coincidence).

**Why it matters**: Putting files in /var/www/ will not work; they must be in /var/www/html/. Questions often ask where to place index.html or why a site is blank.

---

### Trap 5: Samba SWAT Overwrites smb.conf
**The Trap**: Question: "You use SWAT (web interface at port 901) to add a share. Later, you manually edit smb.conf. What happens?"

**The Wrong Answer**: "SWAT updates its interface to match your changes"

**The Right Answer**: SWAT rewrites the entire smb.conf from its database, overwriting your manual edits. Always backup smb.conf before using SWAT, and avoid mixing SWAT and manual edits.

**Why it matters**: SWAT is deprecated and dangerous. Best practice is to edit smb.conf directly, validate with `testparm`, and restart Samba.

---

### Trap 6: sendmail.cf is Generated, Not Edited Directly
**The Trap**: Question: "You need to change Sendmail configuration. Where do you edit?"

**The Wrong Answer**: "Edit /etc/mail/sendmail.cf directly"

**The Right Answer**: Edit /etc/mail/sendmail.mc (macro config), then run `sudo make -C /etc/mail` to generate sendmail.cf. Sendmail reads .cf, but .cf is machine-generated and human-unfriendly.

**Why it matters**: Manually editing sendmail.cf is error-prone and changes get lost when sendmail.mc is recompiled. Always use the .mc file as the source of truth.
