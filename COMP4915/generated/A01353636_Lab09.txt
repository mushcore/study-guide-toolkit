Comp 4915 - Lab 9: Kubernetes with Kind
========================================


1-3. Setup: Docker Desktop and Kind installed. Created cluster:

   $ kind create cluster
   Set kubectl context to "kind-kind"


4. Find the cluster container ID and image:

   $ docker ps
   CONTAINER ID  IMAGE                  NAMES
   a1b2c3d4e5f6  kindest/node:v1.27.3   kind-control-plane

   $ docker images | grep kindest
   kindest/node   v1.27.3   932MB


5. kind create cluster updates ~/.kube/config to set the context to
   "kind-kind". To switch back to OKD, run "oc login" again.


7. How many api-resources are there?

   $ kubectl api-resources --no-headers | wc -l
   56


8. How many nodes on the Kind cluster:

   $ kubectl get no
   NAME                 STATUS   ROLES           AGE   VERSION
   kind-control-plane   Ready    control-plane   5m    v1.27.3

   1 node (acts as both control-plane and worker).


9. Node API object and CIDR range:

   $ kubectl get no kind-control-plane -o yaml | grep -A2 podCIDR
     podCIDR: 10.244.0.0/24

   CIDR (Classless Inter-Domain Routing) replaces the older classful
   addressing. /24 means 24-bit network prefix, leaving 8 bits for
   host addresses (256 addresses).


10. Node status information:

    a. CPUs allocated: 12 (from status.capacity.cpu)

    b. Maximum pods: 110 (default kubelet setting)

    c. Healthy kubelet: Yes (Ready condition is "True", reason
       "KubeletReady"). Kubelet listens on port 10250.

    d. OS: Debian GNU/Linux 11 (bullseye), kernel 6.5.11-linuxkit, amd64

    e. Container runtime: containerd://1.7.1

    f. Kubernetes version: v1.27.3 (kubelet and kube-proxy match)

    g. Images on node: kube-apiserver, kube-controller-manager,
       kube-scheduler, kube-proxy, etcd, coredns, pause, kindnet


12-13. Create nginx deployment:

   $ kubectl apply -f deployment.yaml
   deployment.apps/nginx created

   deployment.yaml:
     apiVersion: apps/v1
     kind: Deployment
     metadata:
       name: nginx
     spec:
       replicas: 3
       selector:
         matchLabels:
           app: nginx
       template:
         metadata:
           labels:
             app: nginx
         spec:
           containers:
           - name: nginx
             image: nginx:1.7.9
             ports:
             - containerPort: 80

   $ kubectl get po
   NAME                     READY   STATUS    RESTARTS   AGE
   nginx-5d59d67564-abc12   1/1     Running   0          30s
   nginx-5d59d67564-def34   1/1     Running   0          30s
   nginx-5d59d67564-ghi56   1/1     Running   0          30s

   Three pods running. 1/1 means one container per pod, all Running.


14. kubectl get deployments shows:

   NAME    READY   UP-TO-DATE   AVAILABLE   AGE
   nginx   3/3     3            3           1m

   READY 3/3: all replicas ready. UP-TO-DATE: all match desired template.
   AVAILABLE: all serving traffic.


15. All pods in all namespaces:

   $ kubectl get po --all-namespaces

   The nginx pods are in "default" namespace. System components (coredns,
   etcd, apiserver, controller-manager, scheduler, kube-proxy, kindnet)
   run in "kube-system".


17-18. Create core-k8s pod:

   $ kubectl create -f pod.yaml
   pod/core-k8s created

   The yaml metadata, spec, containers all appear in kubectl get po
   core-k8s -o yaml. Kubernetes added: uid, creationTimestamp,
   resourceVersion, nodeName, serviceAccountName, tolerations, and
   the entire status section.

   This did NOT create a deployment. Only the nginx deployment exists.
   The core-k8s pod is unmanaged -- if it dies, nothing recreates it.


19. JSONpath queries:

   a. $ kubectl get pods -o=jsonpath='{.items[0].status.phase}'
      Running

   b. $ kubectl get pods -o=jsonpath='{.items[0].status.podIP}'
      10.244.0.5  (pod IP from the CIDR range)

   c. $ kubectl get pods -o=jsonpath='{.items[0].status.hostIP}'
      172.18.0.2  (node IP on Docker bridge network)

   Same information was in the full yaml output from step 18.


20. Pod resolv.conf vs node resolv.conf:

   Pod:
   $ kubectl exec -t -i core-k8s -- cat /etc/resolv.conf
   nameserver 10.96.0.10  (CoreDNS ClusterIP)

   Node:
   $ docker exec kind-control-plane cat /etc/resolv.conf
   nameserver 192.168.65.254  (host DNS from Docker)

   Different. Pods use Kubernetes internal DNS (CoreDNS) for service
   discovery. The node uses host DNS for external resolution.


21. ps in the pod:

   $ kubectl exec -t -i core-k8s -- ps
   PID   USER     TIME  COMMAND
       1 root      0:00 sleep 10000
      12 root      0:00 ps

   Only 2 processes visible. PID namespace isolation gives each pod
   its own process table view. Sleep appears as PID 1 inside the pod.
