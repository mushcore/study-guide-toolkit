Comp 4915 - Lab 2: Unix File System
====================================

Procedures:
  mkdir ~/Assignments
  touch ~/Assignments/lab2.txt
  chmod 600 ~/Assignments/lab2.txt


1) What commands did you use to create the file, and to change the
   permissions on your file?

     mkdir ~/Assignments
     touch ~/Assignments/lab2.txt
     chmod 600 ~/Assignments/lab2.txt

   mkdir creates the directory. touch creates an empty file. chmod 600 sets
   the permissions to rw------- (read and write for the owner only, no
   permissions for group or others).


2) How are directories different from ordinary files? How can they be
   distinguished using the ls utility?

   A directory is a special file that contains a list of directory entries,
   where each entry is a pair of inode number and filename. An ordinary file
   contains user data (text, binary, etc.).

   Using ls -l, the first character of the permission string distinguishes
   them:
     - d  indicates a directory     (e.g., drwxr-xr-x)
     - -  indicates an ordinary file (e.g., -rw-r--r--)

   You can also use ls -F, which appends a / to directory names, or ls -d
   to list a directory entry itself rather than its contents.


3) Is each of the following an absolute pathname, a relative pathname, or a
   simple filename:

   a) milk_co
      Simple filename. It contains no slashes, so it refers to a file in the
      current working directory.

   b) corresp/business/milk_co
      Relative pathname. It contains slashes but does not start with /, so
      it is relative to the current working directory.

   c) /home/alex
      Absolute pathname. It starts with /, specifying the full path from the
      root of the filesystem.

   d) /home/alex/literature/promo
      Absolute pathname. It starts with / and specifies the complete path
      from root.

   e) ..
      Relative pathname. The .. entry refers to the parent directory of the
      current working directory. It is relative because it depends on where
      you currently are.


4) List the commands you can use to:

   a) Make your home directory the working directory:

        cd
        cd ~
        cd $HOME

      All three change the working directory to your home directory. cd with
      no arguments defaults to $HOME.

   b) Identify the working directory:

        pwd

      pwd (print working directory) displays the absolute pathname of the
      current working directory.


5) If your working directory is /home/alex/assn4, what are two different
   commands that you can use to create a subdirectory named classics?

     mkdir classics
     mkdir /home/alex/assn4/classics

   The first uses a simple filename (relative to the working directory). The
   second uses an absolute pathname.

   To remove classics and its contents:

     rm -r classics

   or equivalently:

     rm -rf classics

   The rm -r command recursively removes the directory and all files and
   subdirectories it contains. You cannot use rmdir if classics is not empty;
   rmdir only removes empty directories. So the sequence would be:

     rm classics/*          (remove all files inside)
     rmdir classics         (remove the now-empty directory)

   or simply rm -r classics to do it in one step.


6) The ls -i command displays a filename preceded by the inode number of the
   file. Write a command to output inode/filename pairs for the files in the
   working directory, sorted by inode number.

     ls -i | sort -n

   ls -i outputs each file with its inode number as the first column. Piping
   to sort -n sorts numerically by the first field (the inode number).


7) What specific command would you use to mount the /boot and / directories
   which are on /dev/sda?

   First create mount points, then mount the partitions. On a typical Linux
   system, /dev/sda1 is /boot and /dev/sda2 (or another partition) is /:

     mkdir /mnt/root
     mkdir /mnt/boot
     mount /dev/sda2 /mnt/root
     mount /dev/sda1 /mnt/boot

   The mount command requires root privileges (use sudo). The mount point
   directories must exist before mounting. The actual partition numbers
   (sda1, sda2, etc.) depend on how the disk was partitioned. You can check
   with fdisk -l /dev/sda to see the partition layout.


8) Use the df utility to answer the following questions:

   a) How many file systems are there on your Linux system?

      Run: df

      A typical Linux system has approximately 5-10 mounted file systems,
      including the root filesystem (/), /boot, /home, and several virtual
      filesystems like tmpfs, devtmpfs, and proc. The exact count depends on
      the system configuration. Count the lines of output from df (minus the
      header line).

   b) What file system stores your home directory?

      Run: df ~

      or: df /home

      This shows which filesystem and device your home directory resides on.
      On many systems, /home is on the root filesystem (/) or on its own
      dedicated partition (e.g., /dev/sda3).

   c) Assuming a) is two or greater, attempt to create a hard link to a file
      on another file system. What error message do you get? What happens if
      you attempt to create a symbolic link instead?

      Hard link attempt:
        ln /boot/vmlinuz ~/vmlinuz-link
        Error: ln: failed to create hard link: Invalid cross-device link

      Hard links cannot cross filesystem boundaries because a hard link is
      just another directory entry pointing to the same inode, and inode
      numbers are only unique within a single filesystem.

      Symbolic link attempt:
        ln -s /boot/vmlinuz ~/vmlinuz-link
        (succeeds without error)

      Symbolic links can cross filesystem boundaries because they store the
      pathname of the target as text, not an inode reference. The symlink
      lives on one filesystem and simply contains the path string pointing
      to a file on another filesystem.


9) Examining /etc/passwd:

   a) What character is used to separate fields in /etc/passwd?

      The colon character ( : )

   b) How many fields are used to describe each user?

      7 fields:
        1. Login name
        2. Encrypted password (or x if shadow passwords are used)
        3. User ID (UID)
        4. Group ID (GID)
        5. GECOS / comment field (user's full name and info)
        6. Home directory
        7. Login shell

   c) How many users are on your system?

      Run: wc -l /etc/passwd

      A typical Linux system has approximately 20-40 entries, most of which
      are system accounts (root, bin, daemon, mail, nobody, etc.) rather
      than real human users.

   d) How many login shells are in use in your system?

      Run: awk -F: '{print $7}' /etc/passwd | sort -u | wc -l

      or: cut -d: -f7 /etc/passwd | sort -u

      Common shells found include /bin/bash, /sbin/nologin, /bin/false,
      and /bin/sh. Typically 3-5 different shells are listed.

   e) Does your system use shadow passwords?

      Run: cat /etc/passwd | head -3

      If the second field contains an "x" rather than an encrypted password
      hash, then the system uses shadow passwords. The actual encrypted
      passwords are stored in /etc/shadow instead. Almost all modern Linux
      systems use shadow passwords.

   f) What are the protections on /etc/passwd? Who is the user and group?

      Run: ls -l /etc/passwd

      Typical output: -rw-r--r-- 1 root root

      Permissions are 644: the owner (root) can read and write; group (root)
      and others can only read. The file must be world-readable because many
      programs need to look up user information. However, it does not contain
      actual passwords (those are in /etc/shadow).

   g) What are the protections on /etc/shadow? Who is the user and group?

      Run: ls -l /etc/shadow

      Typical output: -r-------- 1 root root
      or:              ---------- 1 root root

      Permissions are 000 or 400: only root can read the file (and on some
      systems, the shadow group). It is not readable by ordinary users
      because it contains the actual hashed passwords. The owner is root
      and the group is root (or shadow on some distributions).


10) If /home/jenny/draft and /home/alex/letter are links to the same file
    and the following sequence of events occurs, what will the date be in
    the opening of the letter?

    a) Alex gives the command emacs letter.
    b) Jenny gives the command emacs draft.
    c) Jenny changes the date to January 31, 2004, writes and exits emacs.
    d) Alex changes the date to February 1, 2004, writes and exits emacs.

    The date will be February 1, 2004.

    Because draft and letter are links to the same file (same inode), both
    editors opened the same data. However, each editor loaded a copy of the
    file into its own buffer in memory. Jenny wrote first (January 31), saving
    her changes to the file on disk. Then Alex wrote (February 1), overwriting
    the file on disk with his buffer contents. Since Alex's buffer was loaded
    before Jenny wrote, Alex's version does not include Jenny's change. Alex
    writes last, so his date (February 1, 2004) is what remains on disk.

    e) Suppose you have a file that is linked to a file owned by another
       user. What can you do so that changes to the file are no longer
       shared?

       Copy the file to a new file and then remove the link:

         cp letter letter.tmp
         rm letter
         mv letter.tmp letter

       After this, letter is an independent file with its own inode. Changes
       to letter will no longer affect Jenny's draft (and vice versa),
       because they are no longer linked to the same inode.


11) Are there any subdirectories of the root directory that you cannot search
    from a normal user account? Are there any subdirectories of the root
    directory that you cannot read from a normal user account? Explain.

    Yes. Some directories under / restrict access for normal users:

    Cannot search (no execute permission):
    Directories like /root (the root user's home directory) typically have
    permissions drwx------ (700), which means a normal user cannot search
    (traverse) into them. Without execute permission on a directory, you
    cannot cd into it or access any files within it.

    Cannot read (no read permission):
    The same directories (e.g., /root) also lack read permission for others,
    meaning a normal user cannot list their contents with ls. Other examples
    may include /proc/1 (init process info) or certain directories under
    /etc that are restricted.

    Some directories like /lost+found may have permissions drwx------ (700)
    as well, restricting both read and search access to root only.

    Run: ls -ld /root
    Typical output: drwx------  root root
    This confirms normal users cannot search or read /root.


12) Suppose that a user belongs to a group that has all permissions on a file
    named jobs_list, but the user, as the owner of the file, has no
    permissions. Describe what operations the user can perform on jobs_list.
    What command can the user give to grant all permissions?

    Even though the group has all permissions (rwx), the user as owner has
    no permissions (---). Linux checks permissions in order: owner first,
    then group, then others. Since the user IS the owner, Linux applies the
    owner permissions (---) and stops. The group permissions are never
    checked for the owner. Therefore, the user cannot read, write, or
    execute the file.

    However, the owner of a file can always change its permissions using
    chmod, regardless of the current permission settings:

      chmod u+rwx jobs_list

    or equivalently:

      chmod 700 jobs_list

    After this command, the owner has full read, write, and execute
    permissions. The ability to change permissions is a property of
    ownership, not of the permission bits themselves.


13) Create a file named -x in an empty directory. Explain what happens when
    you try to rename it. How can you rename it?

    Create the file:
      touch -- -x

    or:
      touch ./-x

    The -- tells touch that no more options follow, so -x is treated as a
    filename rather than an option flag.

    Attempting to rename it naively fails:
      mv -x newname
      Error: mv: invalid option -- 'x'

    mv interprets -x as an option flag, not a filename. To rename it, use
    the -- separator or prefix with ./:

      mv -- -x newname

    or:

      mv ./-x newname

    Both methods prevent the shell/command from interpreting -x as an option.


14) Suppose that the working directory contains a single file named andor.
    What error message do you get when you run the following command line?

    a) $mv andor and\/or

       The \ escapes the / character, but / is a path separator that cannot
       be part of a filename in Unix. The shell interprets and\/or as the
       path and/or, meaning a file called "or" inside a directory called
       "and". Since no directory called "and" exists, mv produces the error:

         mv: cannot move 'andor' to 'and/or': No such file or directory

    b) Under what circumstances is it possible to run the previous command
       without producing an error?

       If a directory named "and" exists in the current working directory,
       then mv andor and\/or would successfully move andor into the "and"
       directory and rename it to "or". The result would be the file
       and/or.

       To set this up:
         mkdir and
         mv andor and/or
       This succeeds because the destination directory exists.


15) Is it possible to distinguish a file from a hard link to the file? That
    is, given a filename, can you tell if it was created using an ln command?
    Explain.

    No, it is not possible to distinguish the original file from a hard link
    to it. A hard link is simply another directory entry pointing to the same
    inode. Both the original filename and the hard link are equal -- they are
    both directory entries with the same inode number. There is no "original"
    and "copy"; they are indistinguishable references to the same data.

    The only clue that hard links exist is the link count shown by ls -l.
    If a regular file shows a link count greater than 1, it means there are
    multiple directory entries (hard links) pointing to the same inode. But
    you cannot tell which name was created first.

    You can find all hard links to the same inode using:
      find / -inum <inode_number>

    where the inode number is obtained with ls -i.


16) Create 2 files:

    a) $ man bash > wiener; man bash | col -b > sausage

    b) How many characters are in wiener? How many characters are in sausage?
       Explain any differences.

       Run: wc -c wiener sausage

       The wiener file will be larger than the sausage file. The man command
       formats its output for terminal display using backspace characters
       to create bold and underline effects. For example, to bold the
       letter "A", man outputs A, then backspace (\b), then A again
       (overstriking). To underline "A", man outputs _, then backspace,
       then A.

       wiener contains all of these backspace sequences because the output
       was simply redirected to a file.

       sausage has the backspace sequences stripped out by col -b, which
       removes all backspaces and the characters they would overstrike.
       This results in a smaller, cleaner file with only the visible text.


17) What does col -b do?

    col -b strips (removes) all backspace characters and the characters
    that would be overprinted by them. Man pages use backspace sequences to
    create bold text (character + backspace + character) and underlined text
    (underscore + backspace + character) for terminal display. col -b removes
    these formatting sequences, leaving only the final visible characters.
    This produces a clean plain-text version of the man page.


18) Delete the wiener file created in question 16, and rename the sausage
    file to dog, and create a symbolic link to dog, and call it wiener. What
    commands did you use to do all this?

      rm wiener
      mv sausage dog
      ln -s dog wiener

    rm wiener:       deletes the wiener file
    mv sausage dog:  renames sausage to dog
    ln -s dog wiener: creates a symbolic link named wiener that points to dog

    Verification:
      ls -l wiener
      lrwxrwxrwx 1 user user 3 ... wiener -> dog

    The l at the beginning indicates wiener is a symbolic link, and the
    arrow shows it points to dog. Reading wiener will display the contents
    of dog.
